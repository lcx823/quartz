# 机制-数据一致性控制

在多用户或分布式环境中，为了保证数据的正确性和可靠性，文件系统必须提供数据一致性控制机制。这通常通过**事务 (Transaction)**、**检查点 (Checkpoint)** 和**并发控制 (Concurrent Control)** 来实现。

## 1. 事务 (Transaction)
- **定义**: 用于访问和修改各种数据项的一个程序单位，它是一系列读/写操作的集合。
- **核心特性 - 原子性 (Atomicity)**: 事务中的所有操作要么**全部完成**，要么**一个都不做**。
  - **托付 (Commit)**: 事务成功完成所有操作。
  - **天折 (Abort)**: 事务中任一操作失败，必须撤销已做的所有修改，使数据恢复到事务开始前的状态，这个过程称为**回滚 (Rollback)**。
- **实现**: 通过**事务记录 (Transaction Log)** 来实现。日志中记录了事务的开始、对数据项的修改（旧值和新值）、结束（托付或天折）等信息。系统发生故障时，可以通过日志来执行 `undo` (回滚) 或 `redo` (重做) 操作来恢复数据一致性。

## 2. 检查点 (Checkpoint)
- **目的**: 解决事务日志过长导致恢复时间过长的问题。
- **原理**: 系统周期性地创建一个检查点，执行以下操作：
  1.  将内存中当前的事务日志记录强制写入稳定存储器（磁盘）。
  2.  将内存中所有已修改的数据块强制写入磁盘。
  3.  在日志中写入一个`<checkpoint>`记录。
- **作用**: 系统发生故障后进行恢复时，**只需处理最后一个检查点之后的事务记录**，大大缩短了恢复时间。

## 3. 并发控制 (Concurrent Control)
- **目的**: 在允许多个事务并发执行的环境下，保证事务的**顺序性 (Serializability)**，即并发执行的结果与按某种次序串行执行的结果相同。
- **实现机制 - 锁 (Locking)**:
  - **互斥锁 (Exclusive Lock)**:
    - 一个事务对某对象加上互斥锁后，其他任何事务都不能再对该对象加任何锁。
    - 用于**写**操作。
  - **共享锁 (Shared Lock)**:
    - 一个事务对某对象加上共享锁后，其他事务仍可以对该对象加共享锁，但不能加互斥锁。
    - 用于**读**操作。
  - 这种机制非常类似于[[机制-读者写者问题]]的解决方案。

## 4. 重复数据的一致性
- **场景**: 为了可靠性，系统常常为关键文件或数据结构（如索引节点）保存副本。
- **问题**: 当主副本被修改时，必须保证所有副本都被同步修改。
- **检查方法**:
  - **链接数一致性检查 (UNIX)**: 定期扫描整个目录树，计算每个文件的实际链接
