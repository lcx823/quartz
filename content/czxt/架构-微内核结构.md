# 架构-微内核结构

微内核 (Microkernel) 结构是一种现代的[[架构-操作系统的结构|操作系统架构]]，它将OS的功能进行划分，只将最基本、最核心的功能保留在内核中，而将大部分服务（如文件系统、设备驱动、网络协议栈等）作为运行在用户态的服务器进程来实现。

## 1. 基本概念与特点
1.  **足够小的内核**: 内核只包含现代OS最基本的核心功能，如：
    - 与硬件紧密相关的处理。
    - [[机制-进程通信 (IPC)|进程间通信 (IPC)]]。
    - 低级的[[机制-请求分页存储管理|存储器管理]]。
    - [[机制-多处理机OS的进程调度|线程调度]]与切换。
2.  **基于客户/服务器模式**: 操作系统的大部分功能由一系列独立的服务器进程提供。应用程序（客户）通过微内核提供的IPC机制向服务器请求服务。
3.  **策略与机制分离原则**: 将实现某一功能的具体执行机构（**机制**）放在微内核中，而将实现优化或不同目标的算法（**策略**）放在内核外的服务器中。
4.  **采用面向对象技术**: 利用封装、继承等概念来确保OS的可靠性和易扩展性。

## 2. 优点
- **提高了系统可扩展性**: 增加新功能只需添加新的服务器进程，无需修改内核。
- **增强了系统的可靠性**: 服务器进程运行在用户态，某个服务器的崩溃不会影响到微内核或其他服务器。
- **增强了系统的可移植性**: 只有微内核中与硬件相关的部分需要移植。
- **提供了对分布式系统的支持**: 基于消息传递的IPC机制天然适合分布式环境。

## 3. 缺点
- **性能问题**: 客户与服务器之间的每次交互都需要通过微内核进行，这涉及到多次用户态/内核态的切换以及上下文切换，导致性能相比传统的宏内核（Monolithic Kernel）有所下降。

# 架构-实时系统 (Real-Time System)

实时系统是指系统能及时响应外部事件的请求，在规定的时间内完成对该事件的处理，并控制所有实时任务协调一致地运行。

> 实时系统最主要的特征是将时间作为关键参数，它必须对所接收的某些信号做出“及时”或“实时”的反应。
> —— 第1章 1.2.5节

## 1. 实时任务的类型
- **按周期性**:
  - **周期性实时任务**: 外部设备周期性地发出激励信号，要求系统按指定周期循环执行。
  - **非周期性实时任务**: 没有明显的周期性，但必须关联一个**截止时间 (Deadline)**。
- **按对截止时间的要求**:
  - **硬实时 (Hard Real-Time, HRT) 任务**: 系统**必须**满足任务对截止时间的要求，否则可能出现灾难性后果。例如：武器控制系统、工业控制系统。
  - **软实时 (Soft Real-Time, SRT) 任务**: 任务也关联一个截止时间，但并不严格，偶尔错过任务的截止时间对系统的影响不大。例如：信息查询系统、多媒体系统。

## 2. 与分时系统的特征比较
| 特征 | 实时系统 | [[架构-分时系统]] |
| :--- | :--- | :--- |
| **多路性** | 对多路现场信息进行采集和控制 | 为多个终端用户分时服务 |
| **独立性** | 各控制任务间互不干扰 | 各用户间互不干扰 |
| **及时性** | 以控制对象所要求的截止时间为准，通常为毫秒级甚至更低 | 以人能接受的等待时间为准，通常为秒级 |
| **交互性** | 仅限于访问特定的专用服务程序或命令 | 提供广泛的人机对话服务（文件、数据处理等） |
| **可靠性** | 要求**高度可靠**，通常采取多级容错措施 | 要求可靠，但不如实时系统严格 |
