# 机制-信号量 (Semaphore)

信号量机制是由Dijkstra在1965年提出的一种卓有成效的进程同步工具，用于解决[[./概念-临界资源与临界区|互斥与同步]]问题。

## 1. 整型信号量
- **定义**: 一个用于表示资源数目的整型量S。
- **原子操作**:
  - `wait(S)` (P操作):
    ```
    while (S <= 0); // 忙等
    S--;
    ```
  - `signal(S)` (V操作):
    ```
    S++;
    ```
- **缺点**: `wait` 操作采用**忙等**方式，不符合“让权等待”准则。

## 2. 记录型信号量
为解决忙等问题，引入了记录型信号量，它遵循“让权等待”准则。
- **数据结构**:
  ```c
  typedef struct {
      int value; // 资源计数
      struct process_control_block *list; // 等待队列指针
  } semaphore;

- **原子操作**:

  - wait(S):

    codeC

    ```
    S->value--;
    if (S->value < 0) {
        block(S->list); // 阻塞进程，放入等待队列
    }
    ```

  - signal(S):

    codeC

    ```
    S->value++;
    if (S->value <= 0) {
        wakeup(S->list); // 唤醒等待队列中的一个进程
    }
    ```

  - **value的含义**:

    - value > 0: 表示可用资源的数量。
    - value = 0: 表示无可用资源，无进程等待。
    - value < 0: 其绝对值表示等待队列中进程的数量。

## 3. AND型信号量

- **目的**: 解决一个进程需要同时获取多个资源而可能引发[[./概念-死锁|概念-死锁]]的问题。
- **核心思想**: 将进程所需的所有资源，在一次原子操作中全部分配，或一个都不分配。
- **操作**: Swait(S1, S2, ..., Sn) 和 Ssignal(S1, S2, ..., Sn)。
  - Swait: 当所有信号量 S1 到 Sn 都 >= 1 时，才对它们同时执行 wait 操作。否则，进程阻塞在第一个 Si < 1 的信号量上。

## 4. 信号量集

- **目的**: 对AND信号量的扩展，允许进程一次请求或释放多个单位的资源，并设置资源分配的下限。
- **操作**:
  - Swait(S1, t1, d1; ...; Sn, tn, dn): 当所有 Si >= ti 时，才对每个 Si 执行 Si = Si - di。
  - Ssignal(S1, d1; ...; Sn, dn): 对每个 Si 执行 Si = Si + di。

## 5. 信号量的应用

- **实现进程互斥**:
  - 定义一个互斥信号量 mutex，初值为1。
  - 将临界区代码放入 wait(mutex) 和 signal(mutex) 之间。
- **实现进程同步**:
  - 定义一个同步信号量 S，初值为0。
  - 在必须等待的进程中，执行 wait(S)。
  - 在引发同步事件的进程中，执行 signal(S)。
  - wait 和 signal 操作通常位于不同的进程中。
