# 机制-死锁的检测与解除

这种方法允许系统进入[[./概念-死锁|死锁]]状态，但系统能通过检测算法及时发现，并采取相应措施将系统从死锁中解脱出来。

## 1. 死锁的检测
- **核心思想**: 系统保存资源的请求和分配信息，并提供一个算法来检测是否存在死锁。
- **检测方法**:
  - **资源分配图简化法**:
    1.  在资源分配图中，找出一个既不阻塞又非独立的进程节点 `Pi`（其所有请求边都可以被满足）。
    2.  消去 `Pi` 的所有请求边和分配边，使其成为孤立节点，相当于该进程运行完毕并释放了资源。
    3.  重复此过程，直到无法再找到这样的进程。
  - **死锁定理**: 如果一个资源分配图是**不可完全简化**的，那么系统就处于死锁状态。
- **检测算法**:
  - 类似于[[./机制-银行家算法|机制-银行家算法]]中的安全性算法。
  - 设置工作向量 `Work = Available`，并查找是否存在一个进程 `Pi` 其 `Request_i <= Work`。
  - 若找到，则假定其运行完成并释放资源 `Work = Work + Allocation_i`，标记该进程。
  - 重复此过程，如果最终不能标记所有进程，则系统发生了死锁。

## 2. 死锁的解除
当检测到死锁后，系统需要采取措施进行恢复。
- **主要方法**:
  1.  **抢占资源**: 从一个或多个死锁进程中抢占足够数量的资源，分配给其他死锁进程，以打破循环等待。
  2.  **终止进程**: 终止系统中的一个或多个死锁进程，打破循环链。
- **终止进程的策略**:
  - **终止所有死锁进程**: 方法简单，但代价巨大。
  - **逐个终止死锁进程**: 按照某种顺序逐个终止，直到死锁解除。每次终止后都需要重新进行死锁检测。
- **选择被终止进程的依据（代价最小原则）**:
  - 进程的优先级。
  - 进程已执行的时间和还需执行的时间。
  - 进程已使用的资源和未来需要的资源。
  - 进程是交互式还是批处理式。
