# 机制-读者写者问题

读者-写者问题 (Readers-Writers Problem) 是一个经典的同步问题，描述了多个进程如何共享一个数据对象（如文件、记录）。

- **读者 (Reader)**: 只读取数据，不修改。
- **写者 (Writer)**: 修改数据。
- **约束**:
  1.  允许多个读者同时读取共享对象。
  2.  在任何时候，只允许一个写者写入共享对象。
  3.  当一个写者正在写入时，不允许任何读者读取。

## 1. 利用记录型信号量解决 (读者优先)
这是**第一读者-写者问题**的解法，即读者优先。只要没有写者在写，任何读者都可以进入。

- **信号量和变量定义**:
  ```c
  int readcount = 0;        // 记录当前正在读的读者数量
  semaphore rmutex = 1;     // 互斥信号量，用于保护 readcount
  semaphore wmutex = 1;     // 互斥信号量，用于实现写者与其他进程的互斥

- **读者伪代码**:

  codeC

  ```
  reader() {
      do {
          wait(rmutex);
          readcount++;
          if (readcount == 1) { // 第一个读者进入
              wait(wmutex);     // 阻止写者进入
          }
          signal(rmutex);
  
          // perform read operation
  
          wait(rmutex);
          readcount--;
          if (readcount == 0) { // 最后一个读者离开
              signal(wmutex);    // 允许写者进入
          }
          signal(rmutex);
      } while(TRUE);
  }
  ```

- **写者伪代码**:

  codeC

  ```
  writer() {
      do {
          wait(wmutex);
          // perform write operation
          signal(wmutex);
      } while(TRUE);
  }
  ```

## 2. 问题变种

- **读者优先**: 如上解法。如果读者源源不断，写者可能会**饿死**。
- **写者优先 (第二读者-写者问题)**: 一旦有写者准备好，它将尽快执行写操作。即如果有写者在等待，则后续的读者不能开始读。这种情况下，读者可能会饿死。

## 3. 利用信号量集解决

可以利用[[./机制-信号量|信号量集]]的 Swait(S, t, d) 操作来更优雅地解决此问题，并可以增加对读者数量的限制。
