# 第三章 测试分析与设计

![[./assets/3、测试分析与设计/c3.png|c3.png]]
## 3.1 测试目标和准则

### 3.1.1 测试目标
设计和执行测试的主要原因或目的包括：
- 向风险管理活动提供信息。
- 提供软件系统质量有关信息。
- 评估软件产品是否满足相关利益者的期望。
- 评估缺陷修正（清除）而不带来负面效应。
- 评估软件变更实施而不带来负面效应。
- 评估软件是否完全符合合规性要求。

### 3.1.2 项目的具体测试目标
在具体项目中，测试目标通常细化为：
- 提供哪些质量风险信息。
- 新改动的业务是否正确实现，对已有业务是否有负面影响。
- 是否满足功能性要求和非功能性要求。
- 在测试覆盖率、测试效率上的具体要求。

### 3.1.3 如何确定测试目标
确定测试目标需要分析以下几个维度：
1.  **业务影响**：哪些业务改动，会影响哪些已有业务？
2.  **系统影响**：系统改动会影响哪些系统功能和非功能特性？
3.  **覆盖率要求**：新业务/功能需要多少覆盖？已有业务/功能需要多少回归？
4.  **效率**：如何最大程度提高测试效率？

### 3.1.4 测试进入的准则
测试阶段的开始通常需要满足以下准入条件：
- 清楚了解项目的整体计划框架。
- 完成需求规格说明书评审。
- 技术知识或业务知识的储备。
- 标准环境已准备就绪。
- 技术设计文档已提供。
- 足够的资源（人力、硬件等）。
- 人员组织结构及其责任已确定。

## 3.2 测试需求分析

### 3.2.1 测试需求分析的过程
测试需求分析是一个从整体到局部、从抽象到具体的过程，主要包含以下5个步骤：
1.  **了解项目**：背景、产品价值、待解决的业务问题。
2.  **了解用户**：用户是谁，用户所关心的需求。
3.  **功能分解**：从整体到局部，从上到下逐层分解功能特性，确定**待测功能项**。
4.  **非功能分析**：基于系统的特点，分析质量属性，确定**待测非功能项**。
5.  **优先级分析**：基于特性的价值分析和风险分析，确定**测试项优先级**。

### 3.2.2 启发式测试策略模型 (HTSM)
HTSM (Heuristic Test Strategy Model) 提供了一个测试策略的思考框架：
- **项目环境**：资源、约束、条件。
- **产品元素**：结构、功能、数据、平台、操作、时间。
- **质量标准**：功能性、可靠性、易用性、效率等。
- **测试技术**：分析、设计、执行测试的方法。

> **核心思路**：结合环境、产品和质量标准，选择合适的测试技术，以达到可接受的质量。

### 3.2.3 测试需求分析途径
- **简单化、抽象化**：提取核心逻辑。
- **结构化、层次化**：建立功能树或思维导图。
- **可视化、可观察**：使用图表辅助分析。
- **类比、隐喻**：参考类似系统。
- **收集足够的信息或数据**。

### 3.2.4 常见分析模型比较
| 模型类型 | 特点 | 适用场景 |
| :--- | :--- | :--- |
| **跨职能流程图** | 商业建模标准，复用性强，用户易接受。 | 并行、异步支持较差。 |
| **活动图** | UML建模标准，语义丰富，强调行为流和活动内容。 | 描述复杂的业务逻辑流程。 |
| **时序图** | UML建模标准，强调行为流和协作。 | 技术类系统更常用，分析对象间交互。 |
| **数据流图** | IDEF建模标准，强调数据流，未表示出谁执行。 | 计费类系统最适用。 |

### 3.2.5 明确测试项的优先级
确定优先级的依据：
- **产品质量风险的识别**。
- **功能重要性排序**。
- **风险分析**：使用风险矩阵（可能性 vs 影响）。
- **经验与历史数据**。
- **80/20原则**：$80\%$ 的缺陷往往集中在 $20\%$ 的核心模块。

## 3.3 测试用例构成及其设计

### 3.3.1 什么是测试用例
> **测试用例 (Test Case)**：可以独立进行测试执行的最小单元。它是测试内容的一系列情景，每个情景中必须依靠输入和输出，而对软件的正确性进行判断的测试文档。

**重要性**：
- 它是测试工作的指导和准则。
- 是软件测试质量稳定的根本保障。
- 将测试行为转化为可管理的、具体量化的模式。

### 3.3.2 测试用例设计书写标准
一个标准的测试用例通常包含以下核心要素：
1.  **Test Case ID**：唯一标识符。
2.  **Test Case Description (测试标题)**：简述测试目的。
3.  **Prerequisites (前置条件)**：执行前需满足的状态（如：用户已登录）。
4.  **Test Steps (操作步骤)**：详细的执行动作。
5.  **Test Data (测试数据)**：输入的数据（如：用户名、密码）。
6.  **Expected Result (期望结果)**：预期的系统反应。

其他常见属性还包括：所属模块、优先级、层级、自动化标识等。

### 3.3.3 测试用例设计的颗粒度
测试用例的颗粒度分为**粗颗粒度**和**细颗粒度**。
| 颗粒度 | 描述 | 适用性 |
| :--- | :--- | :--- |
| **粗颗粒度** | 描述较为笼统，例如“输入不同的用户名和口令，其结果要满足设定的要求”。 | 适用于探索性测试或资深测试人员。 |
| **细颗粒度** | 具体到每一个值和对应的具体结果，例如“用户名为空，提示：请输入用户名”。 | 适用于回归测试、自动化测试、新手执行。 |

### 3.3.4 测试用例设计的基本原则
1.  **避免含糊**：Pass/Failed 必须清晰，操作环境和步骤明确。
2.  **抽象归类**：将具有相类似功能的测试用例抽象并归类 $\to$ **数据驱动**的测试用例。
3.  **避免冗长和复杂**：建议操作步骤 $\le 7$ 步，坚持“一个测试用例一个验证点”的原则。

### 3.3.5 良好测试用例的特征
- 可以最大程度地找出软件隐藏的缺陷。
- 可以最高效率的找出软件缺陷。
- 可以最大程度地满足测试覆盖要求。
- 既不过分复杂、也不能过分简单。
- 使软件缺陷的表现可以清楚的判定。
- 不包含重复的测试用例。

## 3.4 测试用例的组织和跟踪

### 3.4.1 测试套件 (Test Suite)
> **测试套件**：由一系列测试用例并与之关联的测试环境组合而构成的集合。

**组织方式**：
1.  **按程序功能模块组织**：如“安装/卸载测试”、“注册测试”、“文件操作测试”。
2.  **按测试用例的类型组织**：如“功能测试”、“界面测试”、“压力测试”、“回归测试”。
3.  **按测试用例的优先级组织**：高优先级先执行。

**应用场合**：
- **回归测试**：创建由改动模块用例构成的套件。
- **分阶段执行**：第一阶段只运行特定平台或核心功能的套件。
- **自动化 vs 手工**：分别建立套件。

### 3.4.2 跟踪测试用例
跟踪是为了回答：“进度如何？”、“通过率多少？”、“阻碍是什么？”。
通常使用**燃尽图**或**S曲线**来跟踪工作量（需执行的测试用例数）随时间的变化。

### 3.4.3 维护测试用例
测试用例的维护是持续改进的过程。需要维护的情况包括：
- 之前的设计不全面或不准确。
- 发现了严重的缺陷，需要补充用例以覆盖。
- 新版本添加了新功能或增强了原有功能。
- 用例描述不规范或有歧义。
- 旧的用例已不再使用（需删除）。

### 3.4.4 测试用例的覆盖率
衡量测试充分性的重要指标：
1.  **测试用例本身**：发现缺陷后补充的测试用例数 / 总的测试用例数。
2.  **需求/功能点覆盖率**：每个功能点都要有测试用例，通常要求平均 $>4 \text{个}/FP$ (Function Point)。
3.  **代码覆盖率**：借助工具跟踪执行过程，了解代码行、类、方法等的覆盖情况。

### 3.4.5 测试用例框架的构成
建立有效、有序的组织结构：
$$
\text{测试目标} \xrightarrow{\text{分解}} \begin{cases} \text{功能性} \to \text{模块1/2/.../N} \to \text{父/子/孙 Case} \\ \text{非功能性} \to \text{容错性/性能/...} \to \text{优先级 高/中/低} \end{cases}
$$
