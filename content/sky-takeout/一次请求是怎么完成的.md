  - 一次请求在这个项目里的路径可以先背成：
  - 浏览器/前端 -> Tomcat -> SpringMVC DispatcherServlet -> 拦截器 -> Controller -> Service -> Mapper/MyBatis -> MySQL
    -> 返回 Result -> JSON 响应前端
  - 这个项目跑在 8080 端口，配置在 sky-server/src/main/resources/application.yml:1

  先拿一个具体请求举例

  - 我建议你脑子里一直盯着这个请求：
  - POST /admin/employee
  - 也就是“新增员工”
  - 入口在 sky-server/src/main/java/com/sky/controller/admin/EmployeeController.java:85
  - 这个请求很好，因为它能串起：
      - 拦截器
      - DTO 绑定
      - Service
      - Mapper
      - AOP 自动填充
      - SQL 执行
      - 统一返回

  第 1 步：请求先到 Tomcat

  - Spring Boot 启动后，内嵌 Tomcat 会监听 8080
  - 启动类在 sky-server/src/main/java/com/sky/SkyApplication.java:11
  - 你可以简单理解为：
  - Tomcat 是门口的服务器，先把 HTTP 请求接进来

  第 2 步：进入 Spring MVC 的总调度器

  - 具体类 DispatcherServlet 不在你项目源码里，它是 Spring MVC 提供的
  - 它的职责你只要记一句：
  - 它是总指挥，负责把请求分发给正确的处理链
  - 它会先找：
      - 这个 URL 对应哪个 Controller 方法
      - 这条请求要不要先经过拦截器
      - 参数怎么绑定
      - 返回值怎么转 JSON

  第 3 步：先匹配 Controller 路由

  - 例如 POST /admin/employee
  - Spring 会找到：
      - 类路径 /admin/employee，在 sky-server/src/main/java/com/sky/controller/admin/EmployeeController.java:28
      - 方法 @PostMapping，在 sky-server/src/main/java/com/sky/controller/admin/EmployeeController.java:85
  - 所以最终会命中：
      - addEmp(@RequestBody EmployeeDTO dto)

  这一步的本质是：

  - Spring 根据注解，把 URL 映射到具体 Java 方法

  第 4 步：在真正进 Controller 前，先走拦截器

  - 拦截器注册在 sky-server/src/main/java/com/sky/config/WebMvcConfiguration.java:41
  - 这里规定了：
      - /admin/** 走管理端拦截器
      - /user/** 走用户端拦截器
  - 所以 POST /admin/employee 会先进：
      - JwtTokenAdminInterceptor，在 sky-server/src/main/java/com/sky/interceptor/JwtTokenAdminInterceptor.java:35

  为什么登录接口不用 token

  - 因为配置里把 /admin/employee/login 排除了
  - 位置在 sky-server/src/main/java/com/sky/config/WebMvcConfiguration.java:44
  - 这就形成两个很典型的路径：
      - 登录请求：不拦截
      - 登录后的业务请求：要拦截

  第 5 步：拦截器 preHandle 做什么

  - 先判断当前请求目标是不是 Controller 方法
      - 不是的话直接放行
      - 代码在 sky-server/src/main/java/com/sky/interceptor/JwtTokenAdminInterceptor.java:38
  - 然后从请求头里拿 token
      - request.getHeader(...)
      - 代码在 sky-server/src/main/java/com/sky/interceptor/JwtTokenAdminInterceptor.java:44
  - 再解析 JWT
      - 代码在 sky-server/src/main/java/com/sky/interceptor/JwtTokenAdminInterceptor.java:49
  - 取出员工 id
      - 代码在 sky-server/src/main/java/com/sky/interceptor/JwtTokenAdminInterceptor.java:50
  - 放进 BaseContext
      - 代码在 sky-server/src/main/java/com/sky/interceptor/JwtTokenAdminInterceptor.java:52

  这一步你要真正理解：

  - 拦截器不是处理业务，它只是确认“你是谁，有没有资格进”

  第 6 步：请求体 JSON 绑定成 DTO

  - 现在轮到参数绑定了
  - addEmp(@RequestBody EmployeeDTO dto) 里的 dto 不是你手动 new 的
  - 是 Spring 自动把前端 JSON 转成 Java 对象
  - 这里依赖消息转换器
  - 这个项目自定义了 Jackson 转换器，在 sky-server/src/main/java/com/sky/config/WebMvcConfiguration.java:108
  - 真正设置的是：
      - MappingJackson2HttpMessageConverter
      - 用自定义 JacksonObjectMapper
      - 位置在 sky-server/src/main/java/com/sky/config/WebMvcConfiguration.java:112
  - 自定义对象映射器在：
      - sky-common/src/main/java/com/sky/json/JacksonObjectMapper.java:25

  为什么要自定义 Jackson

  - 主要是统一处理时间格式
  - 比如 LocalDateTime、LocalDate、LocalTime
  - 位置在 sky-common/src/main/java/com/sky/json/JacksonObjectMapper.java:41
  - 所以这个项目里的“JSON -> DTO”和“对象 -> JSON”不是完全默认的，而是有统一规则的

  第 7 步：进入 Controller

  - 现在请求终于到了：
      - EmployeeController.addEmp(...)
      - 位置 sky-server/src/main/java/com/sky/controller/admin/EmployeeController.java:85
  - Controller 的职责很薄：
      - 接收入参
      - 打日志
      - 调 Service
      - 返回 Result.success()
  - 它不直接写 SQL，也不做复杂校验

  你可以理解为：

  - Controller 只负责“接”和“转”

  第 8 步：进入 Service，执行业务逻辑

  - 代码在 sky-server/src/main/java/com/sky/service/impl/EmployeeServiceImpl.java:69
  - 这里会做：
      - DTO -> Entity 属性复制
      - 设置默认密码
      - 设置状态启用
      - 调 mapper 插入数据库
  - 你会注意到，它没有手动设置 createUser/updateUser
  - 因为这件事被 AOP 接管了

  这里你要开始意识到：

  - Service 是真正的业务组织层
  - Controller 不做复杂逻辑
  - Mapper 不做业务判断
  - 业务逻辑主要放 Service

  第 9 步：调用 Mapper，但真正执行前会先被 AOP 切一刀

  - employeeMapper.insert(employee) 在 sky-server/src/main/java/com/sky/service/impl/EmployeeServiceImpl.java:88
  - 这个 mapper 方法上有：
      - @AutoFill(OperationType.INSERT)
      - 见 sky-server/src/main/java/com/sky/mapper/EmployeeMapper.java:25
  - 因为有这个注解，所以在真正执行 SQL 前，会先进：
      - AutoFillAspect
      - 见 sky-server/src/main/java/com/sky/aspect/AutoFillAspect.java:25

  AOP 在这里做什么

  - 判断是 INSERT 还是 UPDATE
  - 如果是 INSERT，就自动填：
      - createTime
      - updateTime
      - createUser
      - updateUser
  - 其中当前操作人 id 来自：
      - BaseContext.getCurrentId()
      - 见 sky-server/src/main/java/com/sky/aspect/AutoFillAspect.java:51

  这一步很关键，你要从“流程图视角”理解：

  - 请求进入拦截器 -> token 解析出员工 id -> 员工 id 放进 BaseContext
  - 后面 AOP 再从 BaseContext 把它拿出来
  - 所以整个请求链条是串着的，不是孤立的

  第 10 步：MyBatis 执行 SQL

  - AOP 补完字段后，才真正执行 mapper 对应的 SQL
  - 这个插入 SQL 用的是注解方式，不是 XML
  - 位置在 sky-server/src/main/java/com/sky/mapper/EmployeeMapper.java:26
  - SQL 就是：
      - 插入 employee 表

  但不是所有 mapper 都是注解 SQL。
  项目里很多复杂 SQL 是 XML 方式，比如：

  - sky-server/src/main/resources/mapper/EmployeeMapper.xml:6
  - sky-server/src/main/resources/mapper/OrdersMapper.xml:46

  所以你要知道：

  - 这个项目是“接口 + 注解/XML 混合方式”的 MyBatis
  - 简单 SQL 可能写注解
  - 动态 SQL、复杂 SQL 常写 XML

  第 11 步：数据库执行成功，结果层层返回

  - SQL 成功后，mapper 返回
  - service 返回
  - controller 返回 Result.success()
  - 见 sky-server/src/main/java/com/sky/controller/admin/EmployeeController.java:90
  - Result 类在：
      - sky-common/src/main/java/com/sky/result/Result.java:18

  这里要理解一个重要点：

  - Controller 返回的不是字符串 JSON
  - 返回的是一个 Java 对象 Result
  - 然后 Spring 再把它序列化成 JSON

  第 12 步：返回值自动转 JSON

  - 因为 Controller 用的是 @RestController
  - 所以返回对象会自动变成 HTTP 响应体中的 JSON
  - 序列化时还是用前面那个消息转换器：
      - MappingJackson2HttpMessageConverter
      - JacksonObjectMapper
  - 所以一次请求的“进”和“出”都离不开消息转换器：
      - 入参 JSON -> DTO
      - 返回对象 -> JSON

  第 13 步：如果中间抛异常，会怎么走

  - 这也是请求生命周期的一部分
  - 假如 service 里抛了业务异常，或者数据库抛了唯一约束异常
  - 不会直接炸成难看的堆栈给前端
  - 而是被：
      - GlobalExceptionHandler
      - sky-server/src/main/java/com/sky/handler/GlobalExceptionHandler.java:24
  - 它会统一返回：
      - Result.error(...)

  所以你要把异常也纳入请求链理解：

  - 正常路径：Controller -> Service -> Mapper -> Result.success
  - 异常路径：抛异常 -> GlobalExceptionHandler -> Result.error

  你现在可以把“一次请求”完整背成这样

  - 请求先进入 Tomcat 和 SpringMVC 调度器
  - 根据路由找到 Controller 方法
  - 先经过拦截器做 JWT 校验
  - 再通过消息转换器把 JSON 绑定成 DTO
  - Controller 调 Service 执行业务
  - Service 调 Mapper，必要时先经过 AOP 自动填充
  - MyBatis 执行 SQL
  - 成功时返回 Result.success
  - 失败时由全局异常处理器返回 Result.error
  - 最后统一由 Jackson 序列化成 JSON 响应给前端

  再给你两个对照案例，帮助你稳住

  - POST /admin/employee/login
      - 不走 token 拦截
      - 直接参数绑定 -> Controller -> Service -> 查库 -> 签发 token -> 返回
  - POST /admin/employee
      - 先走 token 拦截
      - 解析出员工 id 放入 BaseContext
      - 再进入 Controller / Service / Mapper
      - AOP 自动把当前员工 id 填到 createUser

  这两个一对比，你会更清楚：

  - 为什么登录接口要放行
  - 为什么业务接口要鉴权
  - 为什么 BaseContext 和 AOP 能串起来

  你现在最该抓住的 5 个关键词

  - DispatcherServlet：总调度器
  - Interceptor：进 Controller 前做鉴权
  - HttpMessageConverter：JSON 和 Java 对象互转
  - AOP：横切自动填公共字段
  - GlobalExceptionHandler：统一兜底异常响应

 在这个项目里，一次“需要登录才能访问”的请求，一定会先经过哪两个关键环节，然后才会真正进入业务代码？
 拦截器，参数绑定和消息转换器。先经过 JWT 拦截器做鉴权，再经过参数绑定和消息转换器把请求数据转成 DTO，然后才进入 Controller 和 Service。
