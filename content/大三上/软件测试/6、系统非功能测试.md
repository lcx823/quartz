# 第六章 系统非功能测试

![[./assets/6、系统非功能测试/c6.png|c6.png]]
## 6.1 非功能性的系统测试需求

### 6.1.1 概述
系统非功能性测试旨在验证软件产品符合各种**质量特性**的要求，以满足用户和软件企业自身的非功能性需求。
主要的系统非功能性测试类型包括：
- **性能测试**
- **负载测试**
- **容量测试**
- **安全性测试**
- **兼容性测试**
- **可靠性测试**

### 6.1.2 不同应用的测试需求差异
不同类型的应用系统对非功能性测试的需求侧重点不同：
1. **纯客户端软件**（如播放器、工具软件）：
   - 测试要求最低。
   - 关注占用资源较少（CPU、内存）。
   - 关注多操作系统兼容性（Windows, Linux, Mac）。
2. **纯 Web (B/S) 应用系统**（如门户网站）：
   - 强调**性能**和**可用性**。
   - 需支持高并发访问（7×24小时不间断）。
   - 对安全性有较高要求。
3. **C/S 应用系统**（如邮件系统）：
   - 测试需求接近 Web 系统。
   - 可能面临大量并发用户访问。
   - 对端口、协议限制的控制较易实现。
4. **大型复杂企业级系统**：
   - 涉及面广，集成性强。
   - **系统测试要求最高**。
   - 全面考虑各子系统的协同工作能力。

### 6.1.3 主要测试内容与指标
| 特性 | 描述 | 基本测试方法 |
| :--- | :--- | :--- |
| **性能** | 衡量系统资源占用、响应时间、吞吐量。 | 验证测试、基准测试、规划测试。 |
| **容量** | 系统的最大承受能力（如最大并发用户数、最大数据存储量）。 | 持续加载直到极限。 |
| **可靠性** | 在规定时间和条件下维持正常运行的能力。 | 故障转移测试、长时间运行测试。 |
| **安全性** | 防止非法入侵、数据保护的能力。 | 漏洞扫描、模拟攻击、权限验证。 |
| **兼容性** | 在不同环境（硬件、OS、浏览器）下的运行能力及数据共享能力。 | 跨平台测试、数据兼容性测试。 |

## 6.2 概念辨析：负载、压力与性能测试

### 6.2.1 定义
- **负载测试 (Load Test)**：通过模拟实际负载条件，改变负载大小和方式，发现系统问题。
- **性能测试 (Performance Test)**：在真实环境、特定负载下，获取系统性能指标（如响应时间、资源利用率），验证是否满足需求。
- **压力测试 (Stress Test)**：在**强负载**（如大数据量、大量并发）情况下测试系统的稳定性，查看系统在峰值下的表现及恢复能力。

### 6.2.2 关系
- 负载测试是性能测试的一种技术手段。
- 压力测试可以看作是**高负载下**的负载测试。
- 容量测试也采用负载测试技术来实现。

## 6.3 负载测试技术

### 6.3.1 负载测试过程
1.  **确定关键业务**：分析用户行为，确定关键操作路径。
2.  **制定方案**：确定输入/输出参数。
3.  **准备环境与脚本**：搭建测试环境，开发测试脚本。
4.  **设计场景**：设定负载水平、加载方式（如递增、恒值、动态、峰谷）。
5.  **执行测试**：监控并记录数据（吞吐量、响应时间等）。
6.  **结果分析**：分析瓶颈，调整场景进行下一轮测试。

### 6.3.2 输入参数
- **并发用户数/连接数**。
- **思考时间 (Think Time)**：用户发出请求之间的间隔时间。思考时间为0会导致服务器迅速过载。
- **加载模式**：递增、恒值、动态、整天模式、队列模式等。

### 6.3.3 输出参数（性能指标）
- **响应时间 (Response Time)**：从请求发出到接收到响应的时间。
- **吞吐量 (Throughput)**：单位时间内系统处理的请求数或数据量。
- **并发用户数**：同一时间内的访问用户数。
- **QPS (Queries Per Second)**：每秒查询数。
- **TPS (Transactions Per Second)**：每秒事务数。
- **点击率 (Hits Per Second)**：Web应用特有，每秒HTTP请求数。
- **错误率**：失败业务的概率。
- **资源利用率**：CPU、内存、磁盘IO等的使用情况。

> **注意**：如果一个事务包含 $n$ 个查询接口且不请求其他接口，则 $QPS = n \times TPS$。

## 6.4 性能测试

### 6.4.1 性能测试类型
- **性能验证测试**：验证是否达到预定义的指标。
- **性能基准测试**：获取标准配置下的性能数据作为基准。
- **性能规划测试**：在多种环境下测试，为系统部署提供配置建议。

### 6.4.2 常见性能问题
- 资源泄漏（如内存泄漏）。
- 资源瓶颈（线程池、数据库连接池耗尽）。
- CPU利用率过高（>100%）。
- 线程死锁、阻塞。
- 数据库查询效率低。

## 6.5 压力测试

### 6.5.1 测试类型
- **稳定性压力测试**：高负载下持续运行（如24小时以上）。
- **破坏性压力测试**：不断加载直至系统崩溃，寻找极限。
- **渗入测试 (Soak Test)**：长时间运行以发现内存泄漏、GC问题。
- **峰谷测试 (Peak-rest Test)**：高低负载交替，发现深层问题。

## 6.6 性能测试工具：JMeter

### 6.6.1 JMeter 简介
Apache JMeter 是一款开源的性能测试工具，支持多种协议（HTTP, FTP, JDBC等），可进行负载、压力、功能测试。

### 6.6.2 核心组件
1.  **测试计划 (Test Plan)**：测试的根容器。
2.  **线程组 (Thread Group)**：模拟虚拟用户。配置项包括：
    - **线程数**：虚拟用户数。
    - **Ramp-Up Period**：启动所有线程所需时间。
    - **循环次数**：测试执行次数。
3.  **取样器 (Sampler)**：发送请求（如HTTP请求）。
4.  **逻辑控制器 (Logic Controller)**：控制执行逻辑（如循环、条件判断）。
5.  **配置元件 (Config Element)**：配置公共信息（如HTTP请求默认值、CSV数据文件）。
6.  **定时器 (Timer)**：设置请求间的等待时间（如固定定时器、同步定时器）。
    - **同步定时器**：用于模拟并发（集合点）。
7.  **前置/后置处理器**：请求前后的处理（如关联提取）。
8.  **断言 (Assertion)**：验证响应结果（如响应断言、JSON断言）。
9.  **监听器 (Listener)**：收集和展示结果（如察看结果树、**聚合报告**）。

### 6.6.3 聚合报告关键指标
- **Samples**：样本数。
- **Average**：平均响应时间。
- **Median**：中位数（50%请求的响应时间）。
- **90% Line**：90%请求的响应时间小于该值。
- **Throughput**：吞吐量（TPS）。
- **Error %**：错误率。

## 6.7 兼容性测试

### 6.7.1 测试内容
- **硬件兼容性**。
- **数据兼容性**：
    - **向后兼容**：新版本能使用旧版本数据。
    - **向前兼容**：旧版本能使用新版本数据（较难）。
- **系统版本兼容性**。

### 6.7.2 Web兼容性
重点关注不同浏览器（Chrome, Firefox, IE/Edge, Safari）及其不同版本的表现。

## 6.8 安全性测试

### 6.8.1 常见安全漏洞
- **跨站脚本攻击 (XSS)**：注入恶意脚本窃取信息。
- **SQL注入**：通过恶意SQL语句欺骗数据库。
- **缓冲区溢出**。
- **敏感数据泄露**（未加密存储/传输）。
- **权限控制缺陷**（越权访问）。

### 6.8.2 防范措施
- 输入验证（过滤特殊字符）。
- 参数化查询（防SQL注入）。
- 数据加密（传输和存储）。
- 严格的权限管理。

### 6.8.3 常用工具
- **Burp Suite**：Web攻击集成平台。
- **AppScan**、**AWVS**：漏洞扫描。
- **Wireshark**：网络抓包分析。

## 6.9 容错性测试

### 6.9.1 负面测试
检查系统在异常条件或非法操作下的响应，确保不发生崩溃或执行非法动作。

### 6.9.2 故障转移测试 (Failover Test)
验证系统在发生故障时，能否自动切换到备用系统，恢复数据和功能。
- **场景**：Web服务器宕机、数据库断连等。
- **目标**：验证自动备份、恢复机制的有效性。

## 6.10 可靠性测试
验证软件在规定时间和条件下完成规定功能的能力。
- **核心指标**：**平均无故障时间 (MTBF)**。
- **方法**：通过长时间的压力测试和失效模式分析来评估。
