# 第四章 单元测试和集成测试

![[./assets/4、单元测试和集成测试/c4.png|c4.png]]
## 4.1 单元测试及其方法

### 4.1.1 为何要进行单元测试？
- **代码残留错误率高**：编程与编译运行结束后，每1000行代码中大约残留有2-6个Bug。
- **修复成本随阶段增加**：寻找与修改程序错误的代价占总体开发投资的30% - 60%。尽早发现错误，成本越低（发现和修正都更容易）。
- **质量基石**：单元质量是系统质量的基石，单元测试可以比系统测试测得更彻底。
- **其他价值**：
    - 支持扩展。
    - 引导更好的设计。
    - 支持变化（提供安全网）。
    - 避免回归缺陷。
    - 保证工作的平稳步调。
    - 节省测试时间。
    - 规范行为和记录代码。

### 4.1.2 单元测试定义
> 单元测试就是对已实现的软件最小单元进行测试，以保证构成软件系统的各个单元的质量。

- 强调测试对象的**独立性**。
- 应采用不同方式、从不同角度检验单元内部算法和外部功能实现。

### 4.1.3 单元测试的目标
- **正确实现**：单元模块被正确编码实现，包括功能、性能、安全性等。
- **输入输出**：参数输入是否正确传递和得到保护（容错），输出是否正常。
- **数据完整性**：内部数据能否保持其完整性（变量定义、内存释放、全局变量处理）。
- **覆盖率**：代码行、分支覆盖或MC/DC达到要求（如高于80%或95%）。

### 4.1.4 单元测试的方法
单元测试主要采用**白盒测试**方法，辅以**黑盒测试**方法。
- **黑盒测试 (Black-box Testing)**：数据驱动测试。把程序看作黑盒子，不考虑内部结构，只考察输入/输出、外部事件和条件限制。用于模块、组件等大单元的功能测试。
- **白盒测试 (White-box Testing)**：结构化测试或逻辑驱动测试。基于内部逻辑结构（语句、路径、变量状态）进行测试。用于代码评审、单元程序检验。

### 4.1.5 黑盒测试与白盒测试对比
| 方法 | 别名 | 关注点 | 常用技术 |
| :--- | :--- | :--- | :--- |
| **黑盒测试** | 数据驱动测试 | 功能需求、输入/输出 | 等价类划分、边界值分析、判定表、因果图、正交试验 |
| **白盒测试** | 结构化/逻辑驱动测试 | 内部逻辑、代码结构 | 语句覆盖、判定覆盖、条件覆盖、路径覆盖 |

### 4.1.6 驱动程序与桩程序
- **驱动程序 (Driver)**：模拟被测模块的**上级**模块，调用被测模块。
- **桩程序 (Stub)**：模拟被测模块的**下级**模块，被被测模块调用。
> **桩程序的灵活性**：不应只返回硬编码数值，应根据输入参数返回不同值或抛出异常，以模拟真实场景。

## 4.2 逻辑覆盖法

逻辑覆盖是以程序内部逻辑结构为基础的测试方法。

### 4.2.1 语句覆盖 (Statement Coverage)
> 设计若干测试用例，运行被测程序，使程序中的**每个可执行语句至少被执行一次**。

- **特点**：最弱的覆盖标准。无法发现逻辑错误（如条件判断错误）。

### 4.2.2 判定覆盖 (Decision Coverage) / 分支覆盖
> 设计若干用例，使得程序中**每个判断的取真分支和取假分支至少经历一次**。

- **特点**：比语句覆盖强，但仍可能遗漏条件组合。

### 4.2.3 条件覆盖 (Condition Coverage)
> 设计若干用例，使得每个判断中**每个条件的可能取值（真/假）至少满足一次**。

- **特点**：关注具体条件的取值，但未必能覆盖所有分支（如所有条件取真导致分支取真，所有条件取假导致分支取假，可能遗漏其他分支路径）。

### 4.2.4 判定-条件覆盖 (Condition/Decision Coverage)
> 判定和条件覆盖设计方法的交集。设计足够的测试用例，使得判断条件中的**所有条件可能取值至少执行一次**，同时，**所有判断的可能结果至少执行一次**。

### 4.2.5 条件组合覆盖 (Multiple Condition Coverage)
> 设计足够的测试用例，使得判断中**每个条件的所有可能组合至少出现一次**，并且每个判断本身的判定结果也至少出现一次。

- **特点**：覆盖强度较高，但测试用例数量可能呈指数级增长。

### 4.2.6 修正条件/判定覆盖 (MC/DC)
> 每个判定的所有可能结果至少能取值一次；判定中的每个条件的所有可能结果至少取值一次；**一个判定中的每个条件独立地对结果产生影响**；每个入口和出口至少执行一次。

- **公式**：对于 $n$ 个条件的判定，至少需要 $n+1$ 个测试用例。
- **应用**：广泛用于高可靠性系统（如航空航天软件）。

### 4.2.7 逻辑覆盖方法总结
覆盖强度从弱到强通常为：语句覆盖 < 判定覆盖 $\le$ 条件覆盖 < 判定-条件覆盖 < 条件组合覆盖。MC/DC 是一种高效且严格的覆盖标准。

## 4.3 基本路径法

### 4.3.1 基本路径覆盖定义
> 将程序的流程图转化为程序控制流图，分析环路复杂性，导出**基本可执行路径集合**，设计测试用例确保**每条基本路径至少被执行一次**。

### 4.3.2 设计步骤
1.  **绘制控制流图**：依据代码或流程图绘制。
    - 圆圈表示无分支语句，箭头表示控制流。
    - 复合条件（如 `a OR b`）需拆分为嵌套判断。
2.  **计算圈复杂度 (Cyclomatic Complexity, V(G))**：
    - $V(G) = E - N + 2$ （$E$: 边数, $N$: 节点数）
    - $V(G) = P + 1$ （$P$: 判定节点数）
    - $V(G) = \text{区域数}$
3.  **确定独立路径集合**：
    - 独立路径：至少包含一条在之前的路径中未曾用过的边。
    - 路径数等于圈复杂度。
4.  **设计测试用例**：覆盖每条基本路径。

### 4.3.3 实例分析
以判断闰年为例：
- 代码逻辑：`(year % 4 == 0 && year % 100 != 0) || (year % 400 == 0)`
- 绘制流图，计算圈复杂度（例如 $V(G)=4$）。
- 确定4条独立路径。
- 设计对应输入数据（如 1999, 2000, 1900, 2020）和预期结果。

## 4.4 程序插桩法

### 4.4.1 目标代码插桩
> 向目标代码（二进制代码）插入测试代码，获取运行信息。

- **方式**：
    1.  **未运行插桩**：从头到尾插入，适用于完整系统或仿真。
    2.  **运行时插桩**：向正在运行的程序插入，检测特定时间状态。
- **执行模式**：
    - **即时模式**：修改部分代码副本执行。
    - **解释模式**：测试代码作为解释语言执行。
    - **探测模式**：新指令覆盖旧指令（适用于x86等）。
- **工具**：Pin, DynamoRIO。

### 4.4.2 源代码插桩
> 对源文件进行词法、语法分析后，在确认位置植入探针代码。

- **特点**：针对性强，精确性高。
- **应用**：除法运算检测除0错误等。
- **问题**：代码膨胀（20%-100%）、执行效率下降、HeisenBugs（插桩导致缺陷消失或改变）。

## 4.5 代码评审与分析

### 4.5.1 代码标准和规范
- 参考 Google C++/Java Style Guide 等。
- 价值：改善质量、共享知识、辅导新人、遵守规范、增强协作。

### 4.5.2 代码评审 (Code Review)
- **最佳实践**：
    - 一次检查少于 200-400 行。
    - 速度 300-500 LOC/hour。
    - 时间不超过 60-90 分钟。
    - 使用检查表 (Checklist)。
- **工具支持**：GitHub Pull Requests, Gerrit 等。

### 4.5.3 静态分析
- **关注点**：规范、风格、圈复杂度、信息流复杂度、耦合性。
- **SAST工具**：
    - **SonarLint**：IDE插件，实时分析。
    - **FindBugs** (已停止更新，被SpotBugs/IntelliJ自带分析替代)。
    - **CheckStyle**：代码风格检查。
    - **SourceMonitor**：复杂度检测。
- **SCA工具**：软件成分分析，检查开源第三方库的风险（漏洞、许可证）。

## 4.6 单元测试工具

### 4.6.1 常用工具
- **Java**: JUnit, TestNG
- **C++**: GoogleTest
- **Python**: pytest, unittest
- **Mock工具**: JMockit
- **覆盖率**: JaCoCo (Java), gcov (C/C++)

### 4.6.2 JUnit 框架
- **JUnit 4**：
    - 核心类：`Assert` (断言), `TestCase`, `TestSuite`, `TestResult`, `TestRunner`。
- **JUnit 5**：
    - 架构：`JUnit Platform` + `JUnit Jupiter` + `JUnit Vintage`。
    - **注解 (Annotation)**：`@Test`, `@BeforeEach`, `@AfterAll`, `@ParameterizedTest` 等。
    - **断言**：`Assertions` 类提供 `assertEquals`, `assertTrue` 等方法。

### 4.6.3 代码覆盖率工具
- **JaCoCo**：
    - 基于 Java 二进制码，无需源文件。
    - 采用 **On-the-fly instrumentation** (运行时插桩) 技术，通过 Java Agent 在类加载时修改字节码。
    - 报告：行覆盖、分支覆盖、圈复杂度等。
- **EclEmma**：基于 JaCoCo 的 Eclipse 插件。

## 4.7 集成测试

### 4.7.1 集成模式
- **非渐增式 (Big Bang)**：所有模块一次性集成。难定位错误，不推荐。
- **渐增式**：逐步集成。
    - **自顶向下 (Top-down)**：从主控模块开始，使用桩程序 (Stub)。深度优先或宽度优先。
    - **自底向上 (Bottom-up)**：从底层模块开始，使用驱动程序 (Driver)。
    - **混合策略 (Sandwich)**：上层自顶向下，下层自底向上。
    - **改进的三明治**：保证每个模块得到单独测试。

### 4.7.2 持续集成 (CI)
> 频繁地（每天多次）将代码集成到主干。

- **核心实践**：持续检查、构建、部署、验证、报告。
- **持续测试特点**：
    - 足够快（自动化）。
    - 准确有效（精准测试）。
    - 平滑有序（全流程打通）。
    - 与 CI/CD 集成。
