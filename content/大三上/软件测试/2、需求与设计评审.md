# 第二章 需求与设计评审

![[./assets/2、需求与设计评审/c2.png|c2.png]]
## 2.1 软件评审的方法与技术

### 2.1.1 什么是评审
软件评审是对软件元素或者项目状态的一种评估手段，以确定其是否与计划的结果保持一致，并使其得到改进。
> 软件评审是软件开发重要环节之一，也是测试活动之一，即**静态测试**（需求验证）。

**评审的目的**：
借助需求审查保证用户需求在市场/产品需求文档及其相关文档中得到准确、完整、无歧义的反映，并使各类开发人员在需求理解上达成一致。

**评审的分类**：
*   **技术评审**
*   **文档评审**
*   **管理（流程）评审**

### 2.1.2 评审的方法
按照从“最不正式”到“最正式”的顺序，评审方法主要包括：

1.  **临时评审 (Random review)**
2.  **轮查 (Pass-round)**
3.  **互为评审 (Peer review)**
4.  **走查 (Walk-through)**
5.  **会议评审 (Inspection)**

### 2.1.3 评审会议
**1. 评审会议流程**
评审会议通常遵循以下严谨的流程：
1.  **计划**：制定评审计划。
2.  **全面纵览**：概览待评审内容。
3.  **准备**：参与者进行预读和准备。
4.  **评审**：召开会议进行讨论。
5.  **结果分析**：分析发现的问题。
    *   若不满足要求，则进入“修正问题 -> 跟踪 -> 再次评审”的循环。
6.  **修正问题**。
7.  **跟踪**：确保问题已解决。
8.  **总结报告**：输出结论。
9.  **流程改进建议**。

**2. 评审会议角色**
*   **主持人 (Moderator)**：组织会议，控制进度。
*   **作者 (Author)**：文档或代码的编写者。
*   **记录员 (Recorder)**：记录会议中发现的问题和决议。
*   **列席人员**：相关利益方。
*   **内审员/技术专业人员**：负责具体的审查工作。

### 2.1.4 评审的技术
在评审过程中，常使用以下技术手段：
*   **检查表 (Checklist)**：一种常用的质量保证手段，也是正式技术评审的必要工具。评审过程往往由检查表驱动，能显著提高效率和质量。
*   **场景分析**：多用于需求文档评审，按照用户使用场景对产品/文档进行评审。
*   **头脑风暴**：集思广益，发现潜在问题。

## 2.2 产品需求评审

### 2.2.1 需求评审的重要性
**1. 需求缺陷**
软件缺陷并不只是在编程阶段才产生，需求和设计阶段同样会产生缺陷。
> 在需求阶段引入的缺陷，若未及时发现，随着开发过程的推进，修复成本会呈指数级增长。

**2. 测试需求**
测试目标取决于软件质量需求。在制定测试计划前，必须清楚测试需求并明确优先级。
*   **功能性测试需求**：
    *   根据产品规格说明书检验系统是否满足功能要求。
    *   包括：程序安装/启动、功能逻辑、数据输入输出、异常处理、业务流程状态变化等。
*   **用户界面及其显示要求**：
    *   界面友好程度直接影响满意度（KISS原则：Keep it simple, stupid; Don't make me think）。
    *   包括：布局合理、格式正确、色彩协调、链接有效等。
*   **非功能性需求**：
    *   包括系统性能、安全性、兼容性、扩充性。
    *   **SaaS (软件即服务) 的非功能性需求**：
        *   **性能要求**：系统响应能力。
        *   **可用性**：$7 \times 24$ 不间断服务。
        *   **可伸缩性**：支持扩大用户群体的额外负载。
        *   **安全性要求**：确定潜在威胁及策略。
        *   **可维护性要求**：维护的难易程度。

### 2.2.2 如何理解需求
正确理解需求的过程通常为：
确定用户 $\to$ 了解用户业务 $\to$ 业务流程和逻辑图描述 $\to$ 产品功能的分解或分类 $\to$ 业务需求/操作需求 $\to$ 用例/功能特性/非功能特性。

### 2.2.3 传统需求评审的标准
*   **正确性**
*   **完备性**
*   **易理解性**
*   **一致性**
*   **可行性**
*   **易修改性**
*   **易测试性**
*   **易追溯性**

### 2.2.4 如何评审敏捷需求——用户故事 (User Story)
**1. 用户故事的格式**
> 作为一个<角色>，可以<行为>，以便<目标/价值>。
> As a ..., I want ..., So that ...

**2. 用户故事的六个特性 (INVEST)**
*   **I (Independent)**：独立性
*   **N (Negotiable)**：可协商性
*   **V (Valuable)**：业务价值
*   **E (Estimable)**：可估算
*   **S (Small)**：小（足够小以放入迭代）
*   **T (Testable)**：可测试的

**3. 用户故事的评审要点**
*   区分 Activity 是用户的真正需求还是解决方案。
*   寻找隐藏的假设和约束。
*   **建立验收标准 (Acceptance Criteria, AC)**。
    *   格式示例：`Given <condition>, When <Time/Event>, Then <Results>`

**示例：完善测试性**
*   *场景*：用户希望删除购物车中的商品。
*   *对话*：测试人员 (Tester) 通过询问 PO (Product Owner) 边界情况（如批量删除、误删恢复等）来完善故事细节，可能衍生出新的用户故事（如“恢复删除的物品”、“移入愿望清单”）。

## 2.3 设计审查

### 2.3.1 设计评审解决的问题
设计评审旨在解决以下潜在问题：
*   系统架构设计的不合理。
*   单点失效 (Single-point failure)。
*   数据不完整性、不一致性。
*   缺乏可测试性。
*   具体功能设计的问题。

**设计评审的目标**：
保证需求能在设计中得到准确和完整的表示，即保证系统架构设计和产品功能规格说明书的质量。重点关注：**系统架构、可测试性、系统部署**。

**设计评审方法**：
*   **A**：从高层到底层。
*   **B**：不断从测试角度去问开发。
*   **C**：借助 UML 等建模工具。

### 2.3.2 系统架构设计的评审
系统设计评审标准主要包括技术评审标准（稳定、清晰、合理）和非功能性质量特性（安全、性能、扩展等）。
**关键检查点**：
1.  **单一故障点**：整个系统不应该存在单一故障点。
2.  **故障转移机制**：任何组件故障时，系统能正常工作或快速恢复，服务不中断，数据安全。
3.  **负载平衡**：是否采用冗余组件分流负载。
4.  **关键任务**：保证关键任务设计合理、性能良好且可靠。

### 2.3.3 组件设计的审查
1.  组件自身的**内聚性**、**耦合性**。
2.  组件的功能和接口定义正确，文档描述清楚（含硬件、通信、用户接口）。
3.  确定采用的算法，并有详细的过程性描述。
4.  数据结构、数据流和控制流定义正确。

### 2.3.4 界面设计的评审
重点考察：易懂性、易用性、一致性和规范性、美观与协调性、遵守惯例、独特性、快捷方式、自助功能、错误保护。

### 2.3.5 补充：数据库设计评审
在数据库设计阶段结束后进行，考察概念、逻辑、物理结构设计及数据字典和安全保密设计。

**数据库设计检查表 (部分关键项)**

| 序号 | 检查项目 | 说明 |
| :--- | :--- | :--- |
| 1 | 数据库设计是否满足软件设计的一般要求？ | 需满足需求分析中的数据要求 |
| 2 | 数据库设计是否与其他设计内容一致？ | 与软件架构、模块设计保持一致 |
| 3 | 设计是否充分考虑了新系统与现有系统的关系？ | 接口、兼容性、数据共享 |
| 4 | 是否有工具或方案实现快速导入基础数据？ | 针对来源于其他系统的数据 |
| 5 | 反规范化 (违反 $3NF$) 的设计是否有明确说明和理由？ | 如为了提高查询效率而增加冗余列 |
| 6 | 对反规范化设计部分的数据完整性是否充分考虑？ | 如触发器、过程保证一致性 |
| 7 | 孤立表的设计理由是否充分？ | 避免无用的垃圾表 |
| 8 | 是否对大容量表 ($>1000$万行) 做了必要设计？ | 分区表、分表策略等 |
| 9 | 是否避免深度超过三层的视图？ | 保证查询效率 |
| 10 | 是否遵循统一的命名规范？ | 表、字段、索引命名一致 |
| 11 | 表、列、视图、触发器、过程的注释是否完整？ | 文档化 |
| 12 | 命名是否避免使用数据库的保留字？ | 避免语法冲突 |
| 13 | 数据类型是否存在溢出的可能？ | 如 `integer`, `smallint`, `char` 等 |
| 14 | 数据类型的长度是否保留了未来扩展的余量？ | 身份证升位、电话号码升位等 |
| 15 | 查询条件的列上是否建立了 `NOT NULL` 约束？ | 避免 `NULL` 值导致索引失效或逻辑错误 |
| 16 | 是否谨慎地使用日期型字段？ | 区分日期和日期时间，避免精度问题 |
| 17 | 主键是否采用系统生成的键？ | 推荐自增长列、序列、GUID，而非业务含义键 |
| 18 | 是否尽量避免将可能变动的字段作为主键？ | 主键应稳定 |
| 19 | 外键字段是否建立 `NOT NULL` 约束？ | 避免空值关联问题 |
| 20 | 是否尽量使用数据库约束机制实现数据完整性？ | 如主键、外键、Check约束，而非仅靠应用程序 |
| 21 | 索引是否正确建立在查询操作频繁的表上？ | 针对 `where` 子句、连接列等 |
| 22 | 索引是否尽量避免建立在大容量字段上？ | 如 `memo`, `blob`, 大文本 |
| 23 | 索引是否避免建立在小型数据表上？ | 全表扫描可能更快 |
| 24 | 是否将索引建立在独立的表空间上？ | 提高IO性能 |
