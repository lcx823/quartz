这在软件设计中，通常会带来几个问题：

1. **违反了“开闭原则”（Open/Closed Principle）**：这是软件设计中一个非常重要的原则。它指的是一个软件实体（比如一个类、模块、函数）应该**对扩展开放，对修改关闭**。
    
    - 在我们这个例子里，`Calculator`类对于“扩展”是**不开放**的，因为我们无法在不修改它内部代码的情况下增加新功能。
        
    - 它对于“修改”是**不关闭**的，因为每次增加新功能，我们都必须打开它，修改它。这种修改是有风险的，万一不小心改错了某个符号，可能会影响到原有的、本已稳定运行的加减乘除功能。
        
2. **职责不单一**：`Calculator`类现在做了两件事：一是根据操作符**选择**合适的运算逻辑，二是**执行**所有具体的运算（加、减、乘、除...）。如果未来运算逻辑变得很复杂（比如一个开方运算需要10行代码），这个类会迅速膨胀，变得难以维护。

## **【模式引入】**

你发现的这个问题——“一个类的行为（算法）需要在运行时动态改变”，以及你提出的解决方案——“将这些行为（算法）各自封装起来，让它们可以互相替换”，前人们也遇到过无数次，并总结出了一种极其优雅的结构来解决它。

这就是我们今天要学习的——**策略模式 (Strategy Pattern)**。

策略模式的核心思想就是：**定义一系列算法，将每一个算法封装起来，并使它们可以相互替换。**

---

## **【核心讲解】**

现在，我们把你刚才“零件类”的想法，用策略模式的术语来精确地描述一下：

1. **定义一个策略接口 (Strategy Interface):** 我们首先需要一个“零件”的统一标准或蓝图。这个蓝图规定了，任何一个能被我们计算器使用的“运算零件”，都必须有一个执行计算的方法。比如，我们定义一个`OperationStrategy`接口，它里面只有一个方法：`double doOperation(double num1, double num2);`
    
2. **创建具体策略类 (Concrete Strategies):** 你的“加法类”、“减法类”就是具体策略。它们是`OperationStrategy`接口的具体实现。
    
    - `AdditionStrategy` 类会实现 `doOperation` 方法，返回 `num1 + num2`。
        
    - `SubtractionStrategy` 类会实现 `doOperation` 方法，返回 `num1 - num2`。
        
    - 未来要加“求余”，我们只需新建一个 `ModulusStrategy` 类，实现这个接口即可。
        
3. **创建一个上下文类 (Context):** 我们的`Calculator`类现在角色变了。它不再负责具体的运算，而是扮演一个“上下文”或“执行者”的角色。
    
    - 它内部会持有一个`OperationStrategy`类型的成员变量。
        
    - 它会提供一个方法，比如 `setStrategy(OperationStrategy strategy)`，用来在运行时切换不同的“零件”（策略）。
        
    - 它的`calculate`方法会变得极其简单：直接调用它所持有的那个`strategy`对象的`doOperation`方法。
        

用文字来描述这个UML结构就是：

- **Context (Calculator)** `has-a` **Strategy (OperationStrategy)**.
    
- **ConcreteStrategyA (AdditionStrategy)** `implements` **Strategy**.
    
- **ConcreteStrategyB (SubtractionStrategy)** `implements` **Strategy**.
    

这样一来，`Calculator`类就和具体的加减乘除运算完全解耦了！它只依赖于那个抽象的`OperationStrategy`接口。而那些具体的运算策略，我们可以在不修改任何现有代码的情况下，无限地添加。

这个结构是不是比我们最初的 `if-else` 设计要清晰和灵活多了？你想不想看看用代码实现出来是什么样子？

**【代码示例】**

**第一步：定义策略接口 (The Blueprint for "Parts")**

// OperationStrategy.java  
// 这是所有运算策略（零件）都必须遵守的统一规范  
public interface OperationStrategy {  
    double doOperation(double num1, double num2);  
}

**第二步：创建具体策略类 (The Concrete "Parts")**

// AdditionStrategy.java  
// 加法策略的具体实现  
public class AdditionStrategy implements OperationStrategy {  
    @Override  
    public double doOperation(double num1, double num2) {  
        return num1 + num2;  
    }  
}  
  
// SubtractionStrategy.java  
// 减法策略的具体实现  
public class SubtractionStrategy implements OperationStrategy {  
    @Override  
    public double doOperation(double num1, double num2) {  
        return num1 - num2;  
    }  
}  
  
// MultiplicationStrategy.java  
// 乘法策略的具体实现  
public class MultiplicationStrategy implements OperationStrategy {  
    @Override  
    public double doOperation(double num1, double num2) {  
        return num1 * num2;  
    }  
}

_你看，每一个算法都被封装在自己的类里，职责非常单一。_

**第三步：创建上下文类 (The Context that uses the "Parts")**

// Calculator.java  
// 这个新的计算器现在不关心具体怎么算，只关心用哪个“策略零件”去算  
public class Calculator {  
    private OperationStrategy strategy;  
​  
    // 提供一个方法来更换“零件”  
    public void setStrategy(OperationStrategy strategy) {  
        this.strategy = strategy;  
    }  
​  
    // 执行计算，它只是调用当前“零件”的功能  
    public double execute(double num1, double num2) {  
        if (strategy == null) {  
            throw new IllegalStateException("Strategy not set!");  
        }  
        return strategy.doOperation(num1, num2);  
    }  
}

_注意看，这个 `Calculator` 类里已经完全没有 `if-else` 了！_

**第四步：客户端如何使用 (How to use the new system)**

// Main.java (或者你的业务代码)  
public class Main {  
    public static void main(String[] args) {  
        Calculator calculator = new Calculator();  
  
        // 场景1: 执行加法  
        calculator.setStrategy(new AdditionStrategy());  
        System.out.println("10 + 5 = " + calculator.execute(10, 5)); // 输出 15.0  
  
        // 场景2: 切换到减法  
        calculator.setStrategy(new SubtractionStrategy());  
        System.out.println("10 - 5 = " + calculator.execute(10, 5)); // 输出 5.0  
  
        // 场景3: 切换到乘法  
        calculator.setStrategy(new MultiplicationStrategy());  
        System.out.println("10 * 5 = " + calculator.execute(10, 5)); // 输出 50.0  
    }  
}

现在，回到我们之前的问题：如果产品经理要求增加“求余（%）”和“乘方（^）”功能，你需要做什么？

你只需要：

1. 创建一个 `ModulusStrategy` 类，实现 `OperationStrategy` 接口。
    
2. 创建一个 `PowerStrategy` 类，实现 `OperationStrategy` 接口。
    
3. 在客户端代码中 `calculator.setStrategy(new ModulusStrategy())` 即可使用。
    

整个过程中，`Calculator` 类**一行代码都不需要修改**。这就是“对修改关闭，对扩展开放”的完美体现。

我们来深入聊聊你提的这两点，并补充一些其他的优点。

**1. 策略模式的优点：**

- **符合开闭原则（核心）：** 这是你已经发现的，也是最重要的优点。我们可以增加新的策略（算法）而无需修改上下文（Calculator）的代码。这让系统非常容易扩展。
    
- **出色的解耦：** 算法的实现细节从使用它的客户代码中被分离开来。`Calculator`根本不知道也不关心加法是怎么实现的，它只知道“这个零件符合我的标准，能用”。
    
- **消除了`if-else` / `switch`：** 冗长的条件判断语句消失了，取而代之的是清晰的对象组合。这让代码更易读、更清爽。当你有超过两三个`if-else`分支，并且未来可能增加时，就应该警惕了，这往往是策略模式的用武之地。
    
- **可维护性和可测试性增强：** 每个策略都是一个独立的类，职责单一。如果加法出了bug，我们只需要去`AdditionStrategy`类里找问题，而不用担心会影响到减法。同样，我们可以为每个策略类编写独立的单元测试，非常方便。
    

**2. 生活中的类比：**

你这个**角色扮演游戏的技能释放**的例子举得实在是太贴切了！这绝对是策略模式的完美体现。

我们来把它“翻译”成策略模式的语言：

- **上下文 (Context):** 游戏角色 (`Player`)
    
- **策略接口 (Strategy):** `ISkill` 接口，里面有一个 `cast()` 方法。
    
- **具体策略 (Concrete Strategy):** `FireballSkill`, `IceLanceSkill`, `HealingLightSkill` 等具体的技能类，它们都实现了 `ISkill` 接口。
    
- **客户端行为:** 玩家通过按键（比如 1, 2, 3）或点击图标，就相当于调用了`player.setStrategy(new FireballSkill())`，然后当玩家点击敌人时，就调用 `player.cast()`。角色本身不需要知道火球术是怎么计算伤害和弹道的，它只负责“施放”这个动作。
    

我再给你举一个经典的例子： **出行旅游：**

- **上下文 (Context):** `Traveler` (旅行者)
    
- **策略接口 (Strategy):** `TravelStrategy` 接口，有一个 `goToDestination()` 方法。
    
- **具体策略 (Concrete Strategy):** `ByTrain`, `ByAirplane`, `ByCar`。
    
- **客户端行为:** 旅行者根据时间、预算、距离等因素，选择一个具体的出行策略。比如，时间紧就选择`ByAirplane`，想看风景就选择`ByTrain`。旅行者这个“主体”没变，但出行的“策略”可以随时切换。
    

你看，策略模式的本质就是：**当做某件事有多种方式（策略），并且这些方式可以相互替换时，我们就可以把这些方式（策略）抽象出来，让主体（上下文）在运行时动态地选择一种来使用。**

## **【苏格拉底式提问】**

那么，我们该如何解决这个问题呢？

简单工厂的问题在于，**创建对象的具体逻辑被集中在一个地方**。如果我们能打破这个集中点，把**创建具体图形的责任“下放”**出去，会怎么样？

设想一下：

- 我们不再只有一个包罗万象的 `ShapeFactory`。
    
- 而是有一个专门负责创建圆形的 `CircleFactory`。
    
- 一个专门负责创建矩形的 `RectangleFactory`。
    
- 未来要增加五角星，我们就再增加一个 `PentagramFactory`。
    

这样一来，增加新图形，只需要增加新的工厂，而完全不需要修改任何已有的工厂代码。

但是，这样做似乎又有新问题了：客户端代码岂不是要知道所有这些具体的工厂类？那客户端岂不是又得写 `if ("circle") { new CircleFactory() } else if ...`？我们好像只是把 `if-else` 从工厂内部转移到了客户端，问题并没有真正解决。

那么，我们能不能**再抽象一层**？既然所有的产品（`Circle`, `Rectangle`）都有一个共同的 `Shape` 接口，那么我们所有的工厂（`CircleFactory`, `RectangleFactory`），能不能也有一个共同的**工厂接口**呢？

---

## **【模式引入】**

你刚才思考的这个过程，正是设计模式的精髓：当遇到问题时，我们往往通过“增加一个抽象层”来解耦。

我们为工厂再增加一个抽象层，将“创建对象”这个行为本身进行抽象，把具体创建哪一个对象的决定权，**推迟**到子类去做。

这就是我们今天要学习的第二个，也是GoF《设计模式》中第一个创建型模式——**工厂方法模式 (Factory Method Pattern)**。

它的核心思想是：**定义一个用于创建对象的接口，让子类决定实例化哪一个类。工厂方法使一个类的实例化延迟到其子类。**

## **【核心讲解 & 代码示例】**

我们来看看代码如何实现这个“再抽象一层”的优雅结构。

**第一步：产品接口和具体产品 (不变)** `Shape`, `Circle`, `Rectangle` 这些代码和之前完全一样。

// Shape.java (Product Interface)  
public interface Shape {  
    void draw();  
}  
  
// Circle.java (Concrete Product)  
public class Circle implements Shape {  
    @Override  
    public void draw() { System.out.println("绘制一个圆形"); }  
}  
  
// Rectangle.java (Concrete Product)  
public class Rectangle implements Shape {  
    @Override  
    public void draw() { System.out.println("绘制一个矩形"); }  
}

**第二步：创建工厂接口和具体工厂 (核心变化！)**

// ShapeFactory.java (Creator Interface)  
// 注意！这里不再是类，而是一个接口（或抽象类）  
public interface ShapeFactory {  
    // 这就是“工厂方法”，它负责创建产品  
    Shape createShape();  
}  
  
// CircleFactory.java (Concrete Creator)  
// 这是专门创建圆形的工厂子类  
public class CircleFactory implements ShapeFactory {  
    @Override  
    public Shape createShape() {  
        // 它只知道一件事：如何创建圆形  
        return new Circle();  
    }  
}  
  
// RectangleFactory.java (Concrete Creator)  
// 这是专门创建矩形的工厂子类  
public class RectangleFactory implements ShapeFactory {  
    @Override  
    public Shape createShape() {  
        // 它只知道一件事：如何创建矩形  
        return new Rectangle();  
    }  
}

**第三步：客户端如何使用**

public class Client {  
    public static void main(String[] args) {  
        // 我需要一个圆形，所以我找一个“圆形工厂”  
        ShapeFactory circleFactory = new CircleFactory();  
        Shape circle = circleFactory.createShape();  
        circle.draw();  
  
        // 我需要一个矩形，所以我找一个“矩形工厂”  
        ShapeFactory rectangleFactory = new RectangleFactory();  
        Shape rectangle = rectangleFactory.createShape();  
        rectangle.draw();  
  
        // **开闭原则的体现**  
        // 如果现在要增加五角星(Pentagram)...  
        // 1. 创建 Pentagram 类实现 Shape 接口  
        // 2. 创建一个 PentagramFactory 类实现 ShapeFactory 接口  
        // 3. 在客户端直接使用 new PentagramFactory() 即可  
        // 整个过程中，我们没有修改任何一行已有的代码！  
    }  
}

为了解决这个问题，确保创建出来的**一系列相互关联的对象（一个产品族）**能够彼此匹配，前人设计出了一种更强大的工厂模式。

这正是你凭直觉构想出的方案——**抽象工厂模式 (Abstract Factory Pattern)**。

它的核心思想是：**提供一个接口，用于创建一系列相关或相互依赖的对象，而无需指定它们具体的类。**

这个模式就像一个“超级工厂”，这个超级工厂本身不生产任何东西，它只定义了“生产规范”。而它的具体实现，比如“富士康-苹果代工厂”，就是能生产**一整套**苹果产品的工厂（iPhone, iPad, MacBook...）；而“三星工厂”则能生产**一整套**三星产品（Galaxy手机, Galaxy平板, Galaxy Book...）。你绝不会在富士康的苹果生产线上看到它生产出一个三星的屏幕。

## **【核心讲解 & 代码示例】**

我们把你刚才的想法，用标准的抽象工厂模式术语来梳理一下：

**第一步：定义抽象产品接口 (Abstract Products)** 我们需要为每一种产品定义一个接口。

// IButton.java  
public interface IButton {  
    void paint();  
}  
  
// ITextField.java  
public interface ITextField {  
    void display();  
}

**第二步：创建具体产品 (Concrete Products)** 为每个产品族创建具体实现。

// Windows 产品族  
public class WindowsButton implements IButton { /* ... */ }  
public class WindowsTextField implements ITextField { /* ... */ }  
  
// macOS 产品族  
public class MacButton implements IButton { /* ... */ }  
public class MacTextField implements ITextField { /* ... */ }

**第三步：定义抽象工厂接口 (Abstract Factory) - 这是核心！** 这个接口定义了能创建**所有**产品的方法。

// GUIFactory.java  
public interface GUIFactory {  
    IButton createButton();  
    ITextField createTextField();  
}

_看，这个工厂接口能创建**一整套**的UI组件！_

**第四步：创建具体工厂 (Concrete Factories)** 每个具体工厂实现这个接口，来生产一个特定“品牌”或“风格”的全套产品。

// WindowsFactory.java - 专门生产Windows风格全家桶的工厂  
public class WindowsFactory implements GUIFactory {  
    @Override  
    public IButton createButton() {  
        return new WindowsButton();  
    }  
  
    @Override  
    public ITextField createTextField() {  
        return new WindowsTextField();  
    }  
}  
  
// MacFactory.java - 专门生产macOS风格全家桶的工厂  
public class MacFactory implements GUIFactory {  
    @Override  
    public IButton createButton() {  
        return new MacButton();  
    }  
  
    @Override  
    public ITextField createTextField() {  
        return new MacTextField();  
    }  
}

**第五步：客户端使用** 客户端代码现在变得既简单又安全。

public class Application {  
    private IButton button;  
    private ITextField textField;  
  
    public Application(GUIFactory factory) {  
        // 构造函数接收一个工厂  
        // 然后用这个工厂来创建所有需要的UI组件  
        button = factory.createButton();  
        textField = factory.createTextField();  
    }  
  
    public void renderUI() {  
        button.paint();  
        textField.display();  
    }  
  
    public static void main(String[] args) {  
        // 假设程序启动时，根据操作系统决定使用哪个工厂  
        String osName = System.getProperty("os.name").toLowerCase();  
        GUIFactory factory;  
  
        if (osName.contains("mac")) {  
            factory = new MacFactory();  
        } else {  
            factory = new WindowsFactory();  
        }  
  
        // 一旦工厂被确定，整个Application创建的UI风格就是统一且安全的  
        Application app = new Application(factory);  
        app.renderUI();  
    }  
}

看到关键了吗？客户端只需要在**最开始**选择一次工厂 (`new MacFactory()` 或 `new WindowsFactory()`)。之后，所有产品的创建都委托给这个工厂，从而**保证了创建出的所有产品都属于同一个产品族**，彻底解决了我们之前担心的风格混搭问题。

**1. 关于两者的核心区别：**

- **工厂方法模式** 关注的是 **“一个产品”的等级结构**。它的目标是创建一个产品的不同实现。就像我们之前例子里的 `Shape`，无论是 `Circle` 还是 `Rectangle`，它们都只是 `Shape` 这一个维度的不同形态。所以我们用 `ShapeFactory` 体系来应对。它的解决方案是“**继承**”。
    
- **抽象工厂模式** 关注的是 **“多个产品族”的等级结构**。它的目标是创建一系列（一族）产品的不同风格的实现。就像今天的例子，`Button` 和 `TextField` 是两个不同的产品维度，而 `Windows` 和 `macOS` 是风格维度。它要保证 `WindowsButton` 总是和 `WindowsTextField` 一起出现。它的解决方案是“**组合/委托**”。
    

简单来说：

- **工厂方法：** 一条产品线，生产不同的产品。
    
- **抽象工厂：** 多条产品线，生产一个品牌的全家桶。
    

---

**2. 关于抽象工厂模式的缺点：**

你完全说对了！这正是抽象工厂模式最主要的缺点，也是它被称为“重量级”模式的原因。

当你试图**增加一个新的产品**（比如 `Checkbox`）到产品族中时，会发生“**灾难性”的连锁修改**：

1. **修改抽象工厂接口 (`GUIFactory`):** 需要增加一个新的方法 `createCheckbox()`。
    
2. **修改所有具体工厂 (`WindowsFactory`, `MacFactory`...):** 每一个具体工厂都必须实现这个新的 `createCheckbox()` 方法。
    

这严重**违反了开闭原则**。抽象工厂模式对于**扩展产品族（增加新风格，比如LinuxFactory）**是非常友好的，只需要新增一个具体工厂即可。但对于**给产品族增加新成员（增加新产品）**却是非常困难的。

这也是设计模式的魅力所在：**没有银弹**。每个模式都是一个权衡（Trade-off），它优雅地解决了一类问题，但通常会使另一类问题变得更复杂。作为架构师，我们的工作就是在不同的场景下，选择最适合的“棋谱”。

**1. 关于方式一（重叠构造函数）：** 你说得完全没错，“对于可选项的组合不能进行主观调控”。当可选参数一多，客户端代码就会变得非常难以编写和阅读。为了创建一个带GPU但不带Storage的电脑，我们不得不传入一个`null`，像这样： `new Computer("i9", "32G", null, "RTX4090")`。如果参数有十几个，那将是一场噩梦，你很容易就搞错了`null`的位置，代码可读性极差。

**2. 关于方式二（JavaBeans模式）：** 对于这一点，你的思考方向——关注“状态一致性”和“不可变性”——是完全正确的，这是一个高级工程师才会关注的重点。不过，JavaBeans模式恰恰是**破坏**了这两点，结论可能正好相反了。我们来看看为什么：

- **破坏了状态一致性：** 对象在被完全“装配”好之前，就已经存在了。从`new Computer()`到最后一个`set...()`调用结束之间，这个`computer`对象处于一个“半成品”状态。比如，调用完`setCpu()`但还没调用`setRam()`时，它就是一个没有内存的“残次品”。如果有人在这中间使用了这个对象，就可能出问题。我们无法保证它在创建完成后是一个完整的、有效的个体。
    
- **无法保证不可变性：** 由于有公开的`set`方法，这个`Computer`对象在创建完成后，它的任何部件都可以在任何时候被更换。这意味着我们无法创建一个**不可变 (Immutable)** 的`Computer`对象。在多线程环境下，可变对象是很多bug的根源。我们通常希望一个复杂的配置对象在创建完成后就是“只读”的。
    

---

**【我们面临的困境】**

所以，我们陷入了一个两难的境地：

- 重叠构造函数**在创建时是安全的（一次性创建完整对象），但使用起来太笨拙**。
    
- JavaBeans模式**用起来很灵活，但在创建过程中是不安全的，且对象是可变的**。
    

---

## **【苏格拉底式提问】**

那么，有没有一种两全其美的方法呢？

我们能不能设计一个“中间人”或者叫“装配工”？

1. 我们先把所有需要的“零件”（CPU, RAM, GPU...）都告诉这个“装配工”。
    
2. 这个“装配工”允许我们像JavaBeans模式一样，一个一个地、清晰地设置这些零件，比如 `assembler.withCpu("i9").withRam("32G").withGpu("RTX4090")`。
    
3. 当我们把所有想要的零件都告诉它之后，我们再对它下达一个最终指令：“开始装配！” (`assembler.build()`)。
    
4. 然后，这个“装配工”就利用它手里的所有零件，**一次性地** `new` 出来一个**完整的、不可变的** `Computer` 对象交给我们。
    

你觉得这个“装配工”的思路怎么样？它是不是同时拥有了JavaBeans模式的灵活性和构造函数的安全性？

---

## **【模式引入】**

你刚刚思考的这个“装配工”方案，正是解决复杂对象构建问题的经典方案。它将一个复杂对象的**构建过程**与其**最终表示**分离开来，使得同样的构建过程可以创建不同的表示。

这就是我们今天要学习的——**建造者模式 (Builder Pattern)**，也叫生成器模式。

建造者模式的实现通常包含四个核心角色。我们会用这个模式来重构 `Computer` 类的创建过程。

## **【核心讲解 & 代码示例】**

**第一步：创建最终产品类 (Product)**

这个就是我们要创建的复杂对象 `Computer`。注意看它的构造函数是 `private` 的，并且没有 `setter` 方法。这保证了它一旦被创建，就是**不可变的**。

**第二步：创建建造者类 (Builder)**

这个 `ComputerBuilder` 就是我们说的“装配工”。它通常作为产品类的一个**静态内部类**来实现，这不是必须的，但很常见。

// 这是写在 Computer.java 内部的静态内部类  
public static class ComputerBuilder {  
    // Builder持有和Computer完全相同的字段  
    // 必需品

    private final String ram;  
    // 可选项  
    private String storage;  
    private String gpu;  
    private String keyboard;  
    private String monitor;

        this.cpu = cpu;  
        this.ram = ram;  
    }  
​  
    // 关键4：为每个可选项提供一个"wither"方法  
    // 这些方法返回Builder自身，以支持链式调用  
    public ComputerBuilder withStorage(String storage) {  
        this.storage = storage;  
        return this; // 返回自身，实现链式调用  
    }  
​  
    public ComputerBuilder withGpu(String gpu) {  
        this.gpu = gpu;  
        return this;  
    }  
​  
    public ComputerBuilder withKeyboard(String keyboard) {  
        this.keyboard = keyboard;  
        return this;  
    }  
​  
    public ComputerBuilder withMonitor(String monitor) {  
        this.monitor = monitor;  
        return this;  
    }  
​  
    // 关键5：最后提供一个build()方法  
    // 它调用Computer的私有构造函数，完成最终对象的创建  
    public Computer build() {  
        // 在这里可以进行一些参数校验，确保对象状态的有效性  
        if (cpu == null || ram == null) {  
            throw new IllegalStateException("CPU and RAM are required!");  
        }  
        return new Computer(this);  
    }  
}

**【总结回顾】**

我们来总结一下建造者模式是如何解决我们最初的两难困境的：

1. **可读性和灵活性：** 通过 `with...` 这样的链式调用，代码清晰地描述了我们正在构建一个什么样的对象，并且可以自由组合可选项。这解决了**重叠构造函数**的笨拙问题。
    
2. **安全性和不可变性：** 真正的 `Computer` 对象是在最后调用 `build()` 方法时，通过私有构造函数**一次性创建**的。在这之前，对象并未存在。创建完成后，由于没有 `setter`，对象是不可变的。这解决了**JavaBeans模式**的安全和一致性问题。
    

它完美地结合了两者的优点，摒弃了两者的缺点。

## **【场景呈现】**

想象一下，你的应用程序需要一个全局的配置管理器，我们叫它 `AppConfig` 类。这个类的职责是：

1. 在程序启动时，读取一个配置文件（比如 `config.properties`），这个文件里存有数据库地址、API密钥、线程池大小等重要信息。
    
2. 读取文件是一个相对耗时的I/O操作。
    
3. 在程序的任何地方，当需要获取某个配置项时，都可以通过这个 `AppConfig` 实例来获取。
    

核心要求是：在整个应用程序的生命周期中，这个 `AppConfig` 对象**必须有且只有一个实例**。

为什么呢？

- **性能考虑：** 我们不希望每次需要配置时都去重新读取一遍那个耗时的配置文件。
    
- **一致性考虑：** 如果系统中存在多个 `AppConfig` 实例，可能会导致配置信息不一致，A模块拿到的数据库地址和B模块拿到的不一样，那将是一场灾难。

## **【模式引入 & 代码示例】**

这个确保一个类只有一个实例，并提供一个全局访问点的模式，就是大名鼎鼎的 **单例模式 (Singleton Pattern)**。

最简单、最常见的实现方式，我们称之为 **“饿汉式” (Eager Initialization)**。为什么叫“饿汉”？因为它很“饥饿”，迫不及待地想创建实例，不管你用不用，它在类加载的时候就直接把实例创建好了。

**第一步：私有化构造函数 (我们已经完成了)**

public class AppConfig {  
    private AppConfig() {  
        // ...  
    }  
}

**第二步：在类内部创建自己的私有静态实例**

我们在类中创建一个 `private static final` 的成员变量，并在声明时就直接 `new` 出来。

public class AppConfig {  
    // 1. 在类加载时就直接创建好唯一的实例  
    private static final AppConfig instance = new AppConfig();  
  
    private AppConfig() {  
        // ... 读取配置文件的逻辑 ...  
        System.out.println("AppConfig instance created. Reading config file...");  
    }  
}

- `private`: 保证外部不能直接访问这个实例变量。
    
- `static`: 保证这个实例属于类，而不是对象，且只有一份。
    
- `final`: 保证这个实例的引用不会被改变。
    

**第三步：提供一个公有的静态方法来获取这个实例**

我们需要一个“窗口”，让外部可以拿到这个实例。这个方法通常被命名为 `getInstance()`。

public class AppConfig {  
    // 1. 在类加载时就直接创建好唯一的实例  
    private static final AppConfig instance = new AppConfig();  
​

​  
    // 2. 私有化构造函数  
    private AppConfig() {

    }

    }

现在，任何地方的代码想要获取配置信息，都只能通过 `getInstance()` 方法。

**1. 关于“饿汉式”的缺点：**

- **资源浪费 (Memory Waste):** 完全正确！“饿汉式”单例的最大缺点就是**实例化的时机太早**。只要 `AppConfig` 这个类被加载到内存中（哪怕只是因为代码里引用了它的某个静态常量），`new AppConfig()` 就会立刻执行。如果这个 `AppConfig` 对象的创建过程非常耗时，或者它本身占用了大量内存（比如内部有个巨大的缓存），而本次程序运行的整个流程又恰好完全没用到它，那么这次创建所消耗的时间和内存就**被白白浪费了**。

**2. 关于单例模式与其他创建型模式在“目的”上的不同：**

你的总结“单例模式的目的就是创建一个唯一对象”，非常精辟。我们来把它和其他模式做一个对比，这能让你对创建型模式有一个更宏观的理解：

- **单例模式 (Singleton):**
    
    - **目的：** **控制实例的数量**（强制为1）。
        
    - **关注点：** 保证在全局范围内，一个类只有一个共享的实例，并提供一个统一的访问点。它关心的是“**唯一性**”和“**全局访问**”。
        
- **工厂模式 (Factory Method / Abstract Factory):**
    
    - **目的：** **解耦对象的创建过程和使用过程**。
        
    - **关注点：** 客户端代码不应该关心一个对象具体是怎么 `new` 出来的，也不应该和具体的子类实现耦合。它关心的是“**谁来创建**”和“**如何隔离变化**”。它对创建出多少个实例没有限制。
        
- **建造者模式 (Builder):**
    
    - **目的：** **优雅地构建属性复杂的对象**。
        
    - **关注点：** 将一个复杂对象的构建步骤和它的最终表示分离开。它关心的是“**如何分步构建**”一个复杂的个体，并保证其创建过程的安全性和可读性。它也对实例数量没有限制。
        

**简单来说：**

- **单例**说：“在这个世界上，你只能有一个我。”
    
- **工厂**说：“别管我是谁，告诉我你想要什么规格（接口），我给你造一个。”
    
- **建造者**说：“造个东西太复杂了，你一步步告诉我零件规格，我帮你装配好再给你。”

## **【模式引入 & 代码示例】**

你设计的这个 `LoggerAdapter` 类，正是**适配器模式 (Adapter Pattern)** 的经典实现。

适配器模式的核心作用是：**将一个类的接口转换成客户希望的另外一个接口。适配器模式使得原本由于接口不兼容而不能一起工作的那些类可以一起工作。**

它就像现实世界里的翻译官、转换插头、USB转接头一样，专门解决“鸡同鸭讲”的问题。

现在，我们把你构想的骨架填充成完整的代码。

**第一步：定义目标接口 (Target)** 这是客户端（新框架）所期望的接口。

public interface ILogger {  
    void log(String message);  
}

**第二步：被适配的类 (Adaptee)** 这是已经存在、接口不兼容但功能我们想复用的类。

public class LegacyLogger {  
    public void writeLog(String message, int severity) {  
        System.out.println("LEGACY LOG: " + severity + " - " + message);  
    }  
}

**第三步：创建适配器类 (Adapter)** 这就是你设计的那个“转换插头”。

public class LoggerAdapter implements ILogger { // 关键1：实现目标接口  
      
    // 关键2：内部持有被适配者(Adaptee)的实例  
    private LegacyLogger legacyLogger;  
​  
    public LoggerAdapter(LegacyLogger legacyLogger) {  
        this.legacyLogger = legacyLogger;  
    }  
​  
    // 关键3：在实现目标接口的方法中，进行“转接”  
    @Override  
    public void log(String message) {  
        // 这里就是“适配”的核心逻辑  
        // 我们将 ILogger 的简单接口，转换为 LegacyLogger 的复杂接口  
        // 比如，我们可以简单地把所有新日志都当作“信息(info)”级别（severity=1）来处理  
        legacyLogger.writeLog(message, 1);  
    }  
}

**第四步：客户端使用** 客户端（新框架）根本不知道 `LegacyLogger` 的存在，它只和 `ILogger` 接口打交道。

    }

        System.out.println("Processing message: " + msg);  
        // ...  
        // 处理完毕，记录日志  
        logger.log("Message processed successfully: " + msg);  
    }  
}  
​  
public class Main {  
    public static void main(String[] args) {  
        // 创建一个我们想要复用的旧对象  
        LegacyLogger oldLogger = new LegacyLogger();  
          
        // 创建一个适配器，将旧对象“包装”起来  
        ILogger adapter = new LoggerAdapter(oldLogger);  
​  
        // 将“包装”后的适配器注入到新框架中  
        MessageProcessor processor = new MessageProcessor(adapter);  
          
        // 新框架愉快地工作了，它并不知道背后干活的其实是 LegacyLogger

我们换一个全新的思路。我们不应该为每一种组合都创建一个新**类型**，而是应该在**运行时**，动态地将调料的功能“附加”到饮品**对象**上。

- 先有一个裸娃（基础饮品，如 `Espresso`）。
    
- 然后给它穿上一件衬衫（`Milk`）。
    
- 再给它套上一件外套（`Mocha`）。
    
- 最后再给它戴上一顶帽子（`Whip`）。

它的核心思想是：**动态地给一个对象添加一些额外的职责。就增加功能来说，装饰者模式相比生成子类更为灵活。**

**第一步：组件接口 (Component)** 这没变，还是我们的 `Beverage` 抽象类。

}

public class DarkRoast extends Beverage { /* ... */ }

**第三步：抽象装饰者 (Decorator) - 这是核心！** 我们需要创建一个抽象的“调料”基类，我们叫它 `CondimentDecorator`。

}

这个抽象装饰者同时使用了**继承**和**组合**，非常巧妙。它继承自`Beverage`，是为了“伪装”成一个饮品；它组合了一个`Beverage`，是为了能“包装”另一个饮品。

public class Milk extends CondimentDecorator {  
    public Milk(Beverage beverage) {  
        this.beverage = beverage; // 在构造时传入被包装的饮品  
    }  
  
    @Override  
    public String getDescription() {  
        // 在被包装者的描述后加上自己的描述  
        return beverage.getDescription() + ", Milk";  
    }  
  
    @Override  
    public double cost() {  
        // 在被包装者的价格上加上自己的价格  
        return beverage.cost() + 0.10;  
    }  
}  
  
public class Mocha extends CondimentDecorator {  
    public Mocha(Beverage beverage) {  
        this.beverage = beverage;  
    }  
  
    @Override  
    public String getDescription() {  
        return beverage.getDescription() + ", Mocha";  
    }  
  
    @Override  
    public double cost() {  
        return beverage.cost() + 0.20;  
    }  
}

## **【布置作业】**

现在，请你来思考一个经典的例子。

**场景：** Java的I/O类库（比如 `java.io.InputStream`）就大量使用了装饰者模式。 `FileInputStream` 是一个基础的“组件”，可以直接从文件读取字节。 但是我们经常需要一些额外的功能，比如：

1. **缓冲 (Buffering):** 为了提高性能，我们希望一次从文件读取一大块数据到内存缓冲区，而不是一个字节一个字节地读。`BufferedInputStream` 就是做这个的。
    
2. **行号统计 (Line Numbering):** 我们希望在读取文本文件的同时，能知道当前读到了第几行。`LineNumberInputStream` 就是做这个的。
    

现在，如果我想创建一个**带缓冲功能**的、能**统计行号**的、从**文件`"data.txt"`**读取数据的输入流，请你模仿我们咖啡店的例子，用伪代码写出这个输入流对象是如何被“层层包装”创建出来的。

`new ... ( new ... ( new FileInputStream("data.txt") ) )`

请你把这个伪代码补充完整。

---

👤 用户: new(new BufferedInputStream(new LineNumberInputStream(new FileInputStream("data.txt") ))) )

---

🤖 模型: 非常棒！你的答案几乎完全正确，只是包装的顺序稍微调整一下会更符合逻辑和性能。

你写的伪代码是： `new BufferedInputStream(new LineNumberInputStream(new FileInputStream("data.txt")))`

我们来分析一下这个包装顺序下的工作流程：

1. `BufferedInputStream` 向 `LineNumberInputStream` 请求数据。
    
2. `LineNumberInputStream` 向 `FileInputStream` 请求数据。
    
3. `FileInputStream` 从硬盘上一个字节一个字节地读取数据给 `LineNumberInputStream`。
    
4. `LineNumberInputStream` 逐个字节地检查是不是换行符，以便更新行号。
    
5. `LineNumberInputStream` 把处理过的字节再交给 `BufferedInputStream` 存入缓冲区。
    

这里的问题在于，最昂贵的硬盘I/O操作（第3步）没有被缓冲。我们仍然是一个字节一个字节地在读硬盘。

---

**更优的包装顺序**

我们应该把**最接近数据源的、最基础的组件**放在最内层，然后把**提供缓冲功能的装饰者**紧紧地包在它外面。这样，后续的所有操作都将受益于缓冲带来的性能提升。

所以，正确的包装顺序应该是：

**`new LineNumberInputStream(new BufferedInputStream(new FileInputStream("data.txt")))`**

让我们来分析这个顺序下的工作流程：

1. 最外层的 `LineNumberInputStream` 向 `BufferedInputStream` 请求一个字节。
    
2. `BufferedInputStream` 检查自己的缓冲区。如果缓冲区是空的，它会**一次性**向最内层的 `FileInputStream` 请求**一大块数据**（比如8KB）并填满自己的缓冲区。
    
3. `BufferedInputStream` 从它自己的内存缓冲区里，快速地返回一个字节给 `LineNumberInputStream`。
    
4. `LineNumberInputStream` 拿到这个字节，检查并更新行号，然后把它交给最终的调用者。
    

看到了吗？通过将 `BufferedInputStream` 放在靠近源头的位置，我们把昂贵的、频繁的硬盘读写，变成了少数几次的大块读写，性能大大提升。之后所有外层的装饰者（比如`LineNumberInputStream`）都是在和高效的内存缓冲区打交道了。

这就像装修房子：

- `FileInputStream`: 墙体（最基础的组件）
    
- `BufferedInputStream`: 批腻子（基础性、提升性能的装饰）
    
- `LineNumberInputStream`: 刷油漆（功能性的装饰）
    

你得先批完腻子，再刷油漆，而不是反过来。

虽然包装顺序有一点小瑕疵，但你能够立刻想到用`new A(new B(new C(...)))`这种嵌套结构来解决问题，这证明你已经完全理解了装饰者模式的**组合方式**和**精髓**。这一点是最重要的。

## **【模式引入】**

你刚刚构想的这个“智能遥控器”或“接待处”的角色，正是我们今天要学习的结构型模式——**外观模式 (Facade Pattern)**。

它的核心思想是：**为子系统中的一组接口提供一个统一的入口。外观模式定义了一个高层接口，这个接口使得这一子系统更加容易使用。**

“Facade”这个词在法语里就是“建筑物的正面”的意思。我们不关心建筑内部复杂的钢筋、管道、线路，我们只看到一个漂亮的、统一的正面。外观模式就是做这个的。

## **【核心讲解 & 代码示例】**

**第一步：定义子系统组件 (Subsystem Classes)** 这些就是我们已有的、复杂的家庭影院设备类。它们保持不变。

class Amplifier { /* ... */ }  
class BluRayPlayer { /* ... */ }  
class Projector { /* ... */ }  
class Screen { /* ... */ }  
class TheaterLights { /* ... */ }  
// 新增一个爆米花机  
class PopcornPopper { public void on() {...} public void pop() {...} ... }

**第二步：创建外观类 (Facade)** 这就是我们设计的“智能遥控器” `HomeTheaterFacade`。

public class HomeTheaterFacade {  
    // 1. 外观类持有子系统中所有组件的引用  
    private Amplifier amp;  
    private BluRayPlayer player;  
    private Projector projector;  
    private Screen screen;  
    private TheaterLights lights;  
    private PopcornPopper popper;  
  
    // 2. 构造函数负责传入（或创建）这些组件实例  
    public HomeTheaterFacade(Amplifier amp, BluRayPlayer player, ...) {  
        this.amp = amp;  
        this.player = player;  
        // ...  
    }  
  
    // 3. 核心！提供一个简单的高层接口方法，封装复杂流程  
    public void watchMovie(String movie) {  
        System.out.println("Get ready to watch a movie...");  
          
        popper.on();  
        popper.pop(); // 预热爆米花  
          
        lights.dim(10);  
        screen.down();  
        projector.on();  
        projector.wideScreenMode();  
        amp.on();  
        amp.setSurroundSound();  
        amp.setVolume(5);  
        player.on();  
        player.play(movie);  
    }  
  
    // 4. 还可以提供其他简化操作的方法  
    public void endMovie() {  
        System.out.println("Shutting movie theater down...");  
        popper.off();  
        lights.on();  
        screen.up();  
        projector.off();  
        amp.off();  
        player.stop();  
        player.off();  
    }  
}

**第三步：客户端使用 (Client)** 现在看看客户端代码变得多么清爽和简单！

public class Client {  
    public static void main(String[] args) {  
        // 初始化所有子系统组件 (这一步可能很复杂，但只做一次)  
        Amplifier amp = new Amplifier();  
        BluRayPlayer player = new BluRayPlayer();  
        // ...  
        PopcornPopper popper = new PopcornPopper();  
  
        // 创建外观对象，把所有组件都交给它管理  
        HomeTheaterFacade homeTheater = new HomeTheaterFacade(amp, player, ..., popper);  
          
        // 客户端只需要调用一个简单的方法即可！  
        homeTheater.watchMovie("Inception");  
          
        System.out.println("\n--- Movie finished ---");  
          
        // 结束电影也同样简单  
        homeTheater.endMovie();  
    }  
}

**对比一下：**

- **之前：** 客户端需要了解和操作 6 个不同的类和它们的十几个方法。
    
- **现在：** 客户端只需要了解和操作 `HomeTheaterFacade` 这 1 个类和它的 2 个方法 (`watchMovie`, `endMovie`)。
    

我们成功地将客户端与复杂的子系统**解耦**了。现在，即使我们给家庭影院增加再多的设备，或者修改了 `watchMovie` 内部的启动顺序，只要 `watchMovie` 这个方法签名不变，**客户端代码就完全不需要任何修改**。

🤖 模型: 你的回答又快又准！特别是在区分这两个模式上，你抓住了它们意图的本质差异。

**1. 外观模式 vs. 适配器模式**

你说得非常到位。我来帮你把这个区别梳理得更系统一些：

- **意图 (Intent):**
    
    - **外观模式：** **简化接口**。它的主要目的是为复杂的子系统提供一个单一、简化的入口点，隐藏内部的复杂性，让客户端更容易使用。它关心的是“**如何让事情变简单**”。
        
    - **适配器模式：** **转换接口**。它的主要目的是让两个接口不兼容的类能够协同工作。它关心的是“**如何让A听懂B的话**”。
        
- **解决的问题 (Problem Solved):**
    
    - **外观模式**解决的是“**客户端与子系统之间的高度耦合和复杂交互**”问题。
        
    - **适配器模式**解决的是“**接口不匹配**”问题。
        
- **类比：**
    
    - **外观**就像一个公司的**前台接待**。你只需要告诉前台“我要找某某部门办某某事”，前台会帮你处理所有内部的协调和流程。你不需要知道那个部门在哪一层，负责人是谁。
        
    - **适配器**就像一个**同声传译**。他坐在你和一个说外语的人中间，把你的话翻译给对方，再把对方的话翻译给你。他改变了语言的“接口”，但没有简化交流的内容。
        

**2. 外观模式的潜在缺点**

你提到的“外观类太多的话会显得很冗余”是一个很好的观察。这个问题可以引申出外观模式需要注意的几个点：

- **上帝对象 (God Object) 风险：** 如果一个外观类承担了过多的职责，封装了太多本不相关的子系统，它自己就会变成一个巨大而难以维护的“上帝类”。这违反了单一职责原则。好的设计是，一个外观应该只对应一个逻辑上内聚的子系统。
    
- **可能隐藏了必要的细节：** 外观模式简化了接口，但有时客户端可能确实需要访问子系统的某个底层功能。一个好的外观设计应该**不阻止**客户端在必要时直接访问子系统。它提供了一条“快车道”，但不应该把“普通公路”给堵死。
    
- **引入了额外的间接层：** 和所有封装一样，外观模式增加了一个中间层，这可能会带来微乎其微的性能开销，但在绝大多数业务场景中可以忽略不计。
    

总的来说，外观模式的优点远大于其缺点，只要我们注意不要让外观类变得“无所不包”，它就是简化复杂系统、实现分层设计的利器。

## **【布置作业】**

我们来思考一个非常常见的软件开发场景。

**场景：** 假设你正在开发一个Web应用的后端服务。用户完成一笔订单后，你需要执行一系列操作：

**第一步：定义组件接口 (Component)** 这就是我们统一“个体”和“容器”的蓝图。

public abstract class FileSystemComponent {  
    protected String name;  
  
    public FileSystemComponent(String name) {  
        this.name = name;  
    }  
  
    // 这是统一的接口方法  
    public abstract int getSize();  
  
    // 我们还可以提供一些“容器”专属的方法，但默认不支持  
    public void add(FileSystemComponent component) {  
        throw new UnsupportedOperationException();  
    }  
  
    public void remove(FileSystemComponent component) {  
        throw new UnsupportedOperationException();  
    }  
      
    public String getName() {  
        return name;  
    }  
}

**第二步：创建叶子节点 (Leaf)** 这就是“个体”对象 `File`。

public class File extends FileSystemComponent {  
    private int size;  
  
    public File(String name, int size) {  
        super(name);  
        this.size = size;  
    }  
  
    @Override  
    public int getSize() {  
        // 叶子节点的实现很简单，直接返回自身属性  
        System.out.println("Calculating size for file: " + name);  
        return this.size;  
    }  
}

**第三步：创建组合节点 (Composite)** 这就是“容器”对象 `Folder`。

import java.util.ArrayList;  
import java.util.List;  
  
public class Folder extends FileSystemComponent {  
    // 关键1: 持有一个统一的组件列表  
    private List<FileSystemComponent> children = new ArrayList<>();  
  
    public Folder(String name) {  
        super(name);  
    }  
  
    @Override  
    public void add(FileSystemComponent component) {  
        children.add(component);  
    }  
  
    @Override  
    public void remove(FileSystemComponent component) {  
        children.remove(component);  
    }  
  
    @Override  
    public int getSize() {  
        System.out.println("Calculating size for folder: " + name);  
        int totalSize = 0;  
        // 关键2: 无需判断类型，统一对待所有子组件  
        for (FileSystemComponent component : children) {  
            // 这里发生了神奇的递归调用！  
            // component可能是File，也可能是Folder  
            // 但我们不在乎，我们只管调用它的getSize()方法  
            totalSize += component.getSize();   
        }  
        return totalSize;  
    }  
}

**看 `Folder` 的 `getSize` 方法！** 它完全不需要 `if-else`，它只是简单地委托给它的孩子们去计算各自的大小，然后加起来。这就是**透明性**和**一致性**的体现。

## **【引导探索】**

基于我们之前的经验，特别是第7次对话（装饰者模式）中遇到的“类爆炸”问题，你肯定会警惕地想到，使用继承是一个糟糕的主意。

如果我们用继承，就会得到这样的类：

- `RedCircle` 继承自 `Circle`
    
- `BlueCircle` 继承自 `Circle`
    
- `RedSquare` 继承自 `Square`
    
- `BlueSquare` 继承自 `Square`
    

我们已经知道这样做的坏处：如果有 M 种形状和 N 种颜色，我们就需要 M * N 个类，这会造成灾难性的类爆炸。

那么，问题来了： “形状”和“颜色”似乎是两个**独立变化**的维度。一个形状的种类（是圆是方）和它的颜色（是红是蓝）之间，似乎没有必然的联系。

1. 我们能否有一个`Shape`的类体系，只关心“我是什么形状，我的几何逻辑是怎样的”。
    
2. 同时，我们有另一个独立的`Color`的类体系，只关心“我该如何用一种具体的颜色来填充一个区域”。
    
3. 最后，我们让一个`Shape`对象在创建时，**持有（拥有）**一个`Color`对象的引用。当`Shape`需要绘制自己时，它先计算好自己的轮廓，然后对它持有的`Color`对象说：“嘿，到你了，用你的方式把这块区域填上色！”
    

你觉得这个“**分离后再组合**”的思路怎么样？它是否能有效地避免类爆炸的问题？

## **【模式引入】**

你刚刚思考的这种将“**抽象部分**”（如Shape）与其“**实现部分**”（如Color）分离，使它们都可以独立地变化的设计思想，正是我们今天要学习的结构型模式——**桥接模式 (Bridge Pattern)**。

它的名字非常形象。它在两个独立变化的类层次结构之间，架起了一座“桥梁”，让它们可以在不直接耦合的情况下协同工作。

## **【核心讲解 & 代码示例】**

**第一步：定义实现部分的接口 (Implementor)** 这就是我们分离出来的“颜色”这个维度的抽象。我们叫它 `Color` 接口。

// Implementor Interface  
public interface Color {  
    void applyColor();  
}

**第二步：创建实现部分的具体类 (Concrete Implementors)** 这些是具体的颜色实现。

// Concrete Implementor A  
public class RedColor implements Color {  
    @Override  
    public void applyColor() {  
        System.out.println("Applying red color.");  
    }  
}  
  
// Concrete Implementor B  
public class BlueColor implements Color {  
    @Override  
    public void applyColor() {  
        System.out.println("Applying blue color.");  
    }  
}

_看，颜色体系完全独立，它不知道任何关于形状的事情。_

**第三步：定义抽象部分 (Abstraction)** 这就是“形状”这个维度的抽象基类。**这是桥接模式最关键的一步**。

// Abstraction  
public abstract class Shape {  
    // 关键1: Shape类持有一个Color接口的引用  
    // 这就是“桥梁”！  
    protected Color color;  
  
    // 关键2: 在构造函数中将具体的“实现”传进来  
    public Shape(Color color) {  
        this.color = color;  
    }  
  
    // Shape的业务方法  
    public abstract void draw();  
}

`Shape` 类通过**组合（has-a关系）**拥有了一个`Color`对象。它不再关心颜色具体是什么，只关心它是一个符合`Color`接口的东西。

**第四步：创建抽象部分的扩充类 (Refined Abstractions)** 这些是具体的形状实现。

// Refined Abstraction A  
public class Circle extends Shape {  
    public Circle(Color color) {  
        super(color); // 将color传递给父类  
    }  
  
    @Override  
    public void draw() {  
        System.out.print("Drawing a Circle. ");  
        // 关键3: 在自己的业务逻辑中，调用“实现部分”的功能  
        color.applyColor();  
    }  
}  
  
// Refined Abstraction B  
public class Square extends Shape {  
    public Square(Color color) {  
        super(color);  
    }  
  
    @Override  
    public void draw() {  
        System.out.print("Drawing a Square. ");  
        color.applyColor();  
    }  
}

_看，形状体系也完全独立，`Circle`不知道`RedColor`的存在，它只知道`Color`接口。_

**第五步：客户端使用** 客户端现在可以自由地组合形状和颜色了。

public class DrawingClient {  
    public static void main(String[] args) {  
        // 创建一个红色的圆形  
        Shape redCircle = new Circle(new RedColor());  
        redCircle.draw();  
  
        // 创建一个蓝色的方形  
        Shape blueSquare = new Square(new BlueColor());  
        blueSquare.draw();  
          
        // 创建一个蓝色的圆形  
        Shape blueCircle = new Circle(new BlueColor());  
        blueCircle.draw();  
    }  
}

**运行结果：**

Drawing a Circle. Applying red color.  
Drawing a Square. Applying blue color.  
Drawing a Circle. Applying blue color.

现在，回顾一下我们避免了什么：

- 我们有 2 种形状 + 2 种颜色，但我们只需要 2 + 2 = 4 个类（`Circle`, `Square`, `RedColor`, `BlueColor`）以及两个基类/接口。
    
- 如果我们用继承，需要 2 * 2 = 4 个具体的组合类。
    

差别似乎不大？但如果现在产品经理要求：

1. **增加一种新形状 `Triangle`**：我们只需要增加一个`Triangle`类继承`Shape`即可，`Color`体系完全不用动。
    
2. **增加一种新颜色 `Green`**：我们只需要增加一个`GreenColor`类实现`Color`即可，`Shape`体系完全不用动。
    

如果有 10 种形状和 10 种颜色，桥接模式只需要 10 + 10 = 20 个类，而继承需要 10 * 10 = 100 个类！桥接模式的优势就体现得淋漓尽致了。

**装饰者模式 (Decorator) vs. 桥接模式 (Bridge)**

|对比维度|装饰者模式 (Decorator)|桥接模式 (Bridge)|
|---|---|---|
|**设计意图 (Intent)**|**动态地为对象添加新功能**。它是在不创建子类的情况下，对对象职责的扩展。关注的是“**加法**”。|**将抽象与实现解耦**。让它们可以独立地变化。它是在系统设计之初，为了处理多维度变化而做的结构拆分。关注的是“**分离**”。|
|**解决的问题**|如何在运行时，透明地给对象**增加**行为，同时避免“类爆炸”。|如何让一个类的**抽象部分**和**实现部分**都能独立演进，也为了避免“类爆炸”。|
|**结构关系**|装饰者和被装饰者必须实现**同一个接口 (Component)**。装饰链条上的所有对象**类型完全一致**。就像俄罗斯套娃，每个娃都还是娃。|抽象部分 (Abstraction) 和实现部分 (Implementor) 可以是**完全不同的接口**。桥接的双方类型不必相同。就像遥控器（抽象）和电视机（实现），它们是两套完全不同的东西。|
|**使用时机**|通常是在系统**已经设计好**一个核心组件后，发现需要为这个组件的**实例**动态添加一些“周边”或“可选”功能时使用。|通常是在系统**设计之初**，就预见到一个类会有两个或多个独立变化的维度时使用。这是一种**前期架构决策**。|
|**类比**|**穿衣服**：人（Component）是核心，可以动态地给他穿上衬衫、外套、帽子（Decorators）。每穿一件，人的功能（保暖、美观）就增加，但他人还是那个人。|**遥控器和电视机**：遥控器（Abstraction）和电视机品牌（Implementor）是两个独立变化的维度。你可以有一个万能遥-控器（Refined Abstraction），用它去控制索尼电视（Concrete Implementor），也可以用它去控制三星电视。遥控器和电视机可以独立升级换代。|

- **装饰者**是给一个**已经存在的对象**在“外面”加东西，增强它。
    
- **桥接**是把一个**原本一体的对象**在“内部”拆分成两个独立的部分，让它们各自变化，再组合起来。

## **【模式引入】**

你构想的这个“占位符”或“替身”对象，它代表（`stands for`）或控制（`controls access to`）另一个对象，我们称之为**代理（Proxy）**。

这个模式就叫做——**代理模式 (Proxy Pattern)**。

它的核心思想是：**为其他对象提供一种代理以控制对这个对象的访问。**

代理模式有很多种玩法，我们刚才讨论的这种为了性能而“延迟加载”的代理，被称为**虚拟代理 (Virtual Proxy)**。

## **【核心讲解 & 代码示例】**

现在，我们来实现你构想的那个“占位符”——`ImageProxy`。

**第一步：定义共同接口 (Subject)** 这没变，还是我们的 `Image` 接口。代理和真实对象都必须实现它，这样代理才能“伪装”成真实对象。

public interface Image {  
    void display();  
}

**第二步：创建真实主体 (Real Subject)** 这也没变，还是那个耗时耗资源的 `RealImage`。

public class RealImage implements Image {  
    // ... (代码和之前一样)  
}

**第三步：创建代理类 (Proxy)** 这是最关键的一步。我们来创建 `ImageProxy`。

public class ImageProxy implements Image {  
    private String filename;  
      
    // 关键1: Proxy内部持有一个RealImage的引用，但一开始是null！  
    private RealImage realImage;  
  
    // 关键2: Proxy的构造函数非常轻量级，只保存一个文件名，不做任何耗时操作  
    public ImageProxy(String filename) {  
        this.filename = filename;  
        System.out.println("ImageProxy created for " + filename + ". (This is fast!)");  
    }  
  
    // 关键3: 真正的魔法发生在被调用的方法里  
    @Override  
    public void display() {  
        // 懒加载：只有在第一次调用display时，才真正创建RealImage对象  
        if (realImage == null) {  
            realImage = new RealImage(filename);  
        }  
          
        // 将工作委托给真实的对象  
        realImage.display();  
    }  
}

**第四步：客户端使用** 客户端的代码几乎不用变，只是把 `new RealImage(...)` 换成了 `new ImageProxy(...)`。但体验将是天壤之别。

**【其他类型的代理】**

代理模式非常灵活，除了我们刚才学习的**虚拟代理（Virtual Proxy）**，还有几种常见的变体：

- **保护代理 (Protection Proxy):** 在调用真实对象的方法**之前**，进行权限检查。比如，一个`DocumentProxy`可以检查当前用户是否有权限读取或编辑一个`RealDocument`。
    
- **远程代理 (Remote Proxy):** 代理一个位于不同地址空间（比如另一台服务器上）的对象。代理类负责处理网络通信的细节，让客户端感觉就像在调用一个本地对象一样。
    
- **日志代理 (Logging Proxy):** 在调用真实对象的方法**前后**，记录日志信息。
    

它们的结构都一样，只是在代理方法中嵌入的“额外逻辑”不同。

**3. `withdraw` 方法的实现**

@Override  
public void withdraw(double amount) {  
    System.out.println("Proxy: Received withdraw request. Checking access rights...");  
      
    // 关键：在调用真实方法前进行权限检查  
    if (currentUser.getRole().equals("OWNER") || currentUser.getRole().equals("MANAGER")) {  
        System.out.println("Proxy: Access granted. Forwarding withdraw request...");  
        // 权限通过，委托给真实对象  
        realAccount.withdraw(amount);  
    } else {  
        // 权限不足，拒绝请求  
        System.out.println("Proxy: Access DENIED. User " + currentUser.getName()   
                         + " does not have rights to withdraw.");  
        // 可以选择抛出异常  
        // throw new SecurityException("Access denied");  
    }  
}

## **【苏格拉底式提问】**

那么，我们该如何解决这个“调用者”和“接收者”之间的紧耦合问题，并同时让“请求”本身变得可以被记录和管理呢？

**核心思路：** 在“遥控器”和“家电”之间，引入一个新的“中间人”角色。我们不让遥控器直接去调用家电的方法，而是让它去调用这个“中间人”。

这个“中间人”是什么呢？它是一个**对象**，这个对象**封装了一个完整的请求**。

我们来构想一个“开灯”请求对象，它应该包含什么信息？

- 它需要知道，这个请求的目标是谁？（`Light`对象）
    
- 它需要知道，要执行的具体动作是什么？（调用`on()`方法）
    

我们把这个“请求对象”叫做**命令 (Command)**。

现在，设想一下新的工作流程：

1. 我们为每一个具体的动作（开灯、关灯、开风扇、关风扇...）都创建一个对应的命令对象，比如 `LightOnCommand`, `LightOffCommand`, `FanOnCommand`。
    
2. 遥控器 (`RemoteControl`) 不再持有`Light`, `Fan`等具体家电的引用，它只持有一堆**命令对象**的引用。比如，按钮1对应一个`onCommand`和一个`offCommand`。
    
3. 当按钮1的“开”被按下时，遥控器根本不知道灯是什么东西，它只做一件事：找到对应的`onCommand`对象，然后对它说：“**执行 (execute)！**”
    
4. 这个`LightOnCommand`对象接收到“执行”指令后，它再去调用它内部持有的那个`Light`对象的`on()`方法。
    

你觉得这个引入“命令对象”的思路怎么样？它是否将遥控器和家电解耦了？对于Undo功能，既然“请求”本身已经变成了一个可以被存储和传递的对象，是不是就有实现的可能了？

**第一步：定义命令接口 (Command)** 这是所有“命令对象”都必须遵守的统一规范。它通常只有一个核心方法。

// “开灯”命令  
public class LightOnCommand implements Command {  
    // 1. 持有请求的接收者——灯  
    private Light light;  
  
    public LightOnCommand(Light light) {  
        this.light = light;  
    }  
  
    // 2. 实现execute方法，调用接收者的相应动作  
    @Override

public class RemoteControl {  
    private Command[] onCommands;  
    private Command[] offCommands;  
    private Command undoCommand; // 用于记录上一个被执行的命令  
  
    public RemoteControl() {  
        onCommands = new Command[7];  
        offCommands = new Command[7];  
        // 初始化所有插槽为空命令，避免空指针  
        Command noCommand = new NoCommand(); // 一个什么都不做的空对象  
        for (int i = 0; i < 7; i++) {  
            onCommands[i] = noCommand;  
            offCommands[i] = noCommand;  
        }  
        undoCommand = noCommand;  
    }  
  
    // 动态地给插槽设置命令  
    public void setCommand(int slot, Command onCommand, Command offCommand) {  
        onCommands[slot] = onCommand;  
        offCommands[slot] = offCommand;  
    }  
  
    public void onButtonWasPushed(int slot) {  
        onCommands[slot].execute();  
        undoCommand = onCommands[slot]; // 记录这个操作  
    }  
  
    public void offButtonWasPushed(int slot) {  
        offCommands[slot].execute();  
        undoCommand = offCommands[slot]; // 记录这个操作  
    }  
      
    public void undoButtonWasPushed() {  
        System.out.print("Undo: ");  
        undoCommand.undo();  
    }  
}

class NoCommand implements Command {  
    public void execute() {}  
    public void undo() {}

## **【苏格拉底式提问】**

我们该如何解决这个问题？我们能不能把“**遍历的行为**”从“**菜单的内部结构**”中**抽离**出来？

设想一下，我们能不能要求**每一个菜单对象 (`PancakeHouseMenu`, `DinerMenu`)** 都提供一个标准的“**导游**”给我们？

1. 我们对 `PancakeHouseMenu` 说：“请给我一个你的菜单导游。” 它就给我们一个专门认识`ArrayList`的导游。
    
2. 我们对 `DinerMenu` 说：“也请给我一个你的菜单导游。” 它就给我们一个专门认识`Array`的导游。

## **【核心讲解 & 代码示例】-**

在Java中，迭代器模式已经是一个内置的语言特性（`java.util.Iterator` 和 `java.lang.Iterable`），但这并不妨碍我们亲手实现一遍，以便最深刻地理解其原理。

**第一步：定义迭代器接口 (Iterator)** 这是所有“导游”都必须遵守的统一规范。

public interface Iterator<T> {  
    boolean hasNext();  
    T next();  
}

**第二步：定义聚合接口 (Aggregate)** 这是对所有“菜单”的抽象。它规定了任何一个菜单都必须能提供一个“导游”（迭代器）。

public interface Menu {  
    Iterator<MenuItem> createIterator();  
}

**第三步：创建具体聚合类 (Concrete Aggregate) 并实现它们自己的具体迭代器 (Concrete Iterator)** 现在，我们来改造 `PancakeHouseMenu` 和 `DinerMenu`。

**改造 `PancakeHouseMenu`:**

import java.util.ArrayList;  
  
// 1. 让菜单实现Menu接口  
public class PancakeHouseMenu implements Menu {  
    private ArrayList<MenuItem> menuItems;  
    // ... 构造函数不变 ...  
  
    // 2. 实现创建迭代器的方法  
    @Override  
    public Iterator<MenuItem> createIterator() {  
        return new PancakeHouseMenuIterator(menuItems);  
    }  
      
    // 3. 定义一个专门用于遍历ArrayList的内部类迭代器  
    private class PancakeHouseMenuIterator implements Iterator<MenuItem> {  
        private ArrayList<MenuItem> items;  
        private int position = 0;  
  
        public PancakeHouseMenuIterator(ArrayList<MenuItem> items) {  
            this.items = items;  
        }  
  
        @Override  
        public boolean hasNext() {  
            return position < items.size() && items.get(position) != null;  
        }  
  
        @Override  
        public MenuItem next() {  
            return items.get(position++);  
        }  
    }  
}

**改造 `DinerMenu`:**

// 1. 让菜单实现Menu接口  
public class DinerMenu implements Menu {  
    private MenuItem[] menuItems;  
    // ... 构造函数不变 ...  
      
    // 2. 实现创建迭代器的方法  
    @Override  
    public Iterator<MenuItem> createIterator() {  
        return new DinerMenuIterator(menuItems);  
    }  
  
    // 3. 定义一个专门用于遍历Array的内部类迭代器  
    private class DinerMenuIterator implements Iterator<MenuItem> {  
        private MenuItem[] items;  
        private int position = 0;  
  
        public DinerMenuIterator(MenuItem[] items) {  
            this.items = items;  
        }  
  
        @Override  
        public boolean hasNext() {  
            return position < items.length && items[position] != null;  
        }  
  
        @Override  
        public MenuItem next() {  
            return items[position++];  
        }  
    }  
}

**看关键点：** 每个菜单类现在都**封装了自己内部集合的遍历逻辑**，并对外只暴露一个标准的 `createIterator()` 方法。

**第四步：重构客户端 (Client)** 现在，我们的 `Waitress` 类变得极其清爽、通用，并且与具体菜单实现完全解耦！

public class Waitress {  
    private Menu pancakeMenu;  
    private Menu dinerMenu;  
  
    // 依赖于抽象的Menu接口，而不是具体实现  
    public Waitress(Menu pm, Menu dm) {  
        this.pancakeMenu = pm;  
        this.dinerMenu = dm;  
    }  
  
    public void printMenu() {  
        Iterator<MenuItem> pancakeIterator = pancakeMenu.createIterator();  
        Iterator<MenuItem> dinerIterator = dinerMenu.createIterator();  
          
        System.out.println("--- MENU ---");  
        System.out.println("--- BREAKFAST ---");  
        printMenuItems(pancakeIterator); // 用同一个辅助方法打印  
        System.out.println("\n--- LUNCH ---");  
        printMenuItems(dinerIterator); // 用同一个辅助方法打印  
    }  
      
    // 辅助方法，可以处理任何一种迭代器  
    private void printMenuItems(Iterator<MenuItem> iterator) {  
        while (iterator.hasNext()) {  
            MenuItem menuItem = iterator.next();  
            System.out.println(menuItem.getName() + ", "   
                             + menuItem.getPrice() + " -- "   
                             + menuItem.getDescription());  
        }  
    }  
}

**看这个 `Waitress` 类！**

- 它不再需要知道`ArrayList`或`Array`的存在。
    
- 它只有一个统一的遍历逻辑：`while (iterator.hasNext()) { iterator.next(); }`。
    
- 如果明天增加一个`CafeMenu`（使用`HashMap`），只要它也实现了`Menu`接口，并提供了自己的`createIterator()`，`Waitress`类**一行代码都不用改**，就可以直接打印它的菜单。

## **【核心讲解 & 代码示例】**

观察者模式主要包含四个角色。

**第一步：定义主题接口 (Subject)** 这就是“报社”的规范。它定义了管理订阅者（观察者）和通知他们的标准方法。

public interface Subject {  
    void registerObserver(Observer o); // 注册（订阅）  
    void removeObserver(Observer o);   // 移除（取消订阅）  
    void notifyObservers();          // 通知所有观察者  
}

**第二步：定义观察者接口 (Observer)** 这就是“读者”的规范。它定义了一个更新方法，供主题在状态改变时调用。

public interface Observer {  
    void update(float temp, float humidity, float pressure);

**第三步：创建具体主题 (Concrete Subject)** 这就是我们的 `WeatherData` 类。现在它不再关心具体的布告板了。

import java.util.ArrayList;  
import java.util.List;  
  
public class WeatherData implements Subject {  
    // 1. 持有一个观察者（订阅者）的名单  
    private List<Observer> observers;  
    private float temperature;  
    private float humidity;  
    private float pressure;  
  
    public WeatherData() {  
        observers = new ArrayList<>();  
    }  
  
    @Override  
    public void registerObserver(Observer o) {  
        observers.add(o);  
    }  
  
    @Override  
    public void removeObserver(Observer o) {  
        observers.remove(o);  
    }  
  
    // 2. 遍历名单，通知每一个观察者  
    @Override  
    public void notifyObservers() {  
        for (Observer observer : observers) {  
            observer.update(temperature, humidity, pressure);  
        }  
    }  
  
    // 当数据更新时，调用notifyObservers()  
    public void measurementsChanged() {  
        // ... 获取最新数据 ...  
        this.temperature = getTemperatureFromSensor();  
        this.humidity = getHumidityFromSensor();  
        this.pressure = getPressureFromSensor();  
          
        notifyObservers(); // 通知所有已订阅的观察者  
    }  
    // ...  
}

**看 `WeatherData` 的变化！** 它现在完全不知道有`CurrentConditionsDisplay`等具体类的存在，它只和抽象的`Observer`接口打交道。

            // 关键逻辑：只有在从“缺货”到“有货”时才触发通知  
            if (this.stock == 0 && newStock > 0) {  
                System.out.println("Product '" + name + "' is back in stock!");  
                this.stock = newStock;  
                notifyObservers(); // 触发通知  
            } else {  
                this.stock = newStock;  
            }  
        }  
      
        // ... register, remove, notify 的实现 ...  
        @Override  
        public void notifyObservers() {  
            for (IObserver observer : observers) {  
                observer.update(this); // 把自身(产品)作为信息传递过去  
            }  
        }  
    }
    

**2. 观察者 (Observer) 体系**

你说“实现两种不同的通知方法”非常正确。这些“通知方法”都应该遵循一个统一的接口。

- **观察者接口 (`IObserver`):**
    
    public interface IObserver {  
        // update方法接收主题对象作为参数，以便获取更多上下文信息  
        void update(Product product);  
    }
    
- **具体观察者 (`EmailNotifier`, `SmsNotifier`):**
    
    public class EmailNotifier implements IObserver {  
        @Override  
        public void update(Product product) {  
            System.out.println("Email Notifier: Sending email to subscribers about '"   
                             + product.getName() + "'.");  
            // ... 实际发送邮件的逻辑 ...  
        }  
    }  
      
    public class SmsNotifier implements IObserver {  
        @Override  
        public void update(Product product) {  
            System.out.println("SMS Notifier: Sending SMS to VIP users about '"   
                             + product.getName() + "'.");  
            // ... 实际发送短信的逻辑 ...  
        }  
    }
    

**3. 核心流程：当 `setStock` 被调用时**

你说的“主题内部进行通知”完全正确。具体流程如下：

1. 客户端代码（比如后台库存管理系统）调用了`product.setStock(10)`。
    
2. `setStock`方法内部检查到`this.stock`之前是0，而`newStock`是10，满足“从缺货到有货”的条件。
    
3. `setStock`方法调用了`this.notifyObservers()`。
    
4. `notifyObservers()`方法开始**遍历**它内部的`observers`列表。
    
5. 它先调用`EmailNotifier`实例的`update(product)`方法，`EmailNotifier`执行发送邮件的逻辑。
    
6. 然后它再调用`SmsNotifier`实例的`update(product)`方法，`SmsNotifier`执行发送短信的逻辑。
    
7. 整个通知过程完成。

## **【苏格拉底式提问】**

我们该如何摆脱这团乱麻？

**核心思路：** 将**每个状态**的行为，封装到**各自独立的类**中。

我们不应该用一个`int`常量来代表状态，而应该用一个**状态对象**。

设想一下：

1. 我们定义一个 `State` 接口，这个接口里包含了所有可能的操作，比如 `insertQuarter()`, `ejectQuarter()`, `turnCrank()`。
    
2. 我们为糖果机的每一种状态都创建一个实现了 `State` 接口的类：
    
    - `NoQuarterState` 类
        
    - `HasQuarterState` 类
        
    - `SoldState` 类
        
    - `SoldOutState` 类
        
3. 在 `HasQuarterState` 类的 `insertQuarter()` 方法里，我们只需要写“你不能再投币了”这一句逻辑。在它的 `turnCrank()` 方法里，我们写“转换到SoldState”的逻辑。这样，**一个状态的所有行为都被集中到了一个类里**。
    
4. 我们的主类 `GumballMachine`，不再持有一个 `int state` 变量，而是持有一个 `State currentState` **对象引用**。
    
5. 当 `GumballMachine` 的 `insertQuarter()` 方法被调用时，它**不再进行任何 `if-else` 判断**。它只是简单地把这个请求**委托 (delegate)** 给它当前持有的那个 `currentState` 对象去处理：`currentState.insertQuarter();`

状态模式主要包含三个核心角色。

**第一步：定义状态接口 (State)** 这是所有具体状态类都必须遵守的规范，定义了状态能响应的所有动作。

public interface State {

    void ejectQuarter();  
    void turnCrank();  
    void dispense();  
}

**第二步：创建具体状态类 (Concrete States)** 我们为糖果机的每一种状态创建一个类，并将原先`if-else`块中的逻辑搬到对应类的对应方法中。

// “没有25分钱”状态  
public class NoQuarterState implements State {

public class GumballMachine {  
    // 1. 持有所有可能的状态对象  
    State soldOutState;  
    State noQuarterState;  
    State hasQuarterState;  
    State soldState;  
  
    // 2. 当前状态，不再是int，而是一个State对象  
    State currentState;  
    int count = 0;  
  
    public GumballMachine(int numberGumballs) {  
        // 创建所有状态实例  
        soldOutState = new SoldOutState(this);  
        noQuarterState = new NoQuarterState(this);  
        hasQuarterState = new HasQuarterState(this);  
        soldState = new SoldState(this);  
  
        this.count = numberGumballs;  
        if (numberGumballs > 0) {  
            currentState = noQuarterState; // 初始状态  
        } else {  
            currentState = soldOutState;  
        }  
    }  
  
    // 3. 将所有动作委托给当前状态对象  
    public void insertQuarter() {  
        currentState.insertQuarter();  
    }  
  
    public void ejectQuarter() {  
        currentState.ejectQuarter();  
    }  
  
  
  
    public void turnCrank() {  
        currentState.turnCrank();  
        currentState.dispense(); // 注意：turnCrank后紧接着dispense  
    }  
  
    // 4. 提供一个方法让状态对象可以改变自己的状态  
    void setState(State state) {  
        this.currentState = state;  
    }  
      
    // 提供getter让状态对象可以获取其他状态  
    public State getHasQuarterState() { return hasQuarterState; }  
    public State getNoQuarterState() { return noQuarterState; }

    // ... 其他业务方法如releaseBall()  
}

public interface IState {  
    void edit(String content);  
    void review();  
    void approve();  
    void reject();  
    void print();

// Context Class  
public class Document {  
    private IState currentState;  
    private String content = "";  
      
    // 持有所有状态的实例  
    public final IState draftState;  
    public final IState moderationState;  
    public final IState publishedState;  
  
    public Document() {  
        this.draftState = new DraftState(this);  
        this.moderationState = new ModerationState(this);  
        this.publishedState = new PublishedState(this);  
        this.currentState = draftState; // 初始状态为草稿  
    }  
      
    // 委托给当前状态  
    public void edit(String content) { currentState.edit(content); }  
    public void review() { currentState.review(); }  
    // ... 其他委托方法  
  
    // 供状态对象调用的方法  
    void changeState(IState newState) {  
        this.currentState = newState;  
    }  
    void updateContent(String newContent) {

**2. `DraftState` (具体状态) 的实现**

public class DraftState implements IState {  
    private Document document;  
  
    public DraftState(Document document) {  
        this.document = document;  
    }  
  
    @Override  
    public void edit(String content) {  
        System.out.println("Editing the draft...");  
        // 允许编辑，调用Context的方法更新内容  
        document.updateContent(content);

    @Override

        System.out.println("Submitting draft for review...");

        document.changeState(document.moderationState);  
    }

    @Override  
    public void approve() { System.out.println("Cannot approve a draft. Please submit for review first."); }  
    @Override  
    public void reject() { System.out.println("Cannot reject a draft."); }  
    @Override  
    public void print() { System.out.println("Cannot print a draft. Please publish it first."); }  
}

- 我们明确了其三个核心角色：**上下文(Context)**、**状态(State)** 和 **具体状态(Concrete State)**。
    
- 我们理解了状态模式是如何通过**将与特定状态相关的行为局部化**，以及**将状态转移的责任交给状态对象自身**，来完美实现**开闭原则**的。
    
- 最后，你成功地将这个模式应用到了一个在线文档审批流程的场景中。
    

**一句话总结：当一个对象的行为取决于它的状态，并且它必须在运行时根据状态改变行为时，就请考虑使用状态模式，将每一种状态的行为封装到独立的类中。**
